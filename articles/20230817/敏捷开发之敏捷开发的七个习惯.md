# 1.简介
  

2001年一位敏捷领域的先驱布雷柏克（Bryce Boehm）发表了一篇名为“敏捷宣言”（Agile Manifesto），定义了敏捷开发（Agile Development）的一些基本原则、理念、方法和规范，并将其称为“敏捷过程”（agile process）。从那时起，敏捷开发就经历了一个蓬勃发展的阶段，在全球范围内得到广泛应用。而随着“精益创业”（Lean Startup）、DevOps、云计算等新思潮的出现，越来越多的人开始认识到，敏捷开发仍然是一个非常有效的开发方法论，它可以帮助开发团队更好地响应市场变化，减少开发风险，提升产品质量，还可以适应快速迭代的需求变化。因此，越来越多的公司都开始采用敏捷开发方式进行软件开发工作。
近几年来，随着敏捷开发理论和实践日渐成熟，越来越多的人开始关注敏捷开发的各种理念、规范和实践，特别是《敏捷开发修炼手册》（Agile Principles, Patterns and Practices in Cooperation）这本书，开始成为一个热门话题。除了《敏捷宣言》和《敏捷开发修炼手册》外，还有一本重要的著作——《企业应用程序架构模式》（Patterns of Enterprise Application Architecture）也是大家重点关注的。本文将结合《企业应用程序架构模式》这本书的内容，阐述我认为《敏捷开发的七个习惯》这一系列文章的价值、意义和作用。
《企业应用程序架构模式》这本书由大卫·凡尔纳、马丁·福勒、蒂姆·库克等著名技术作者共同撰写，主要讨论企业级应用架构的设计原则和模式，涉及面向服务的体系结构（SOA）、基于事件驱动的数据管理（EDM）、基于CQRS的应用架构模式等。其中包括的模式和原则很多，本文只选取其中几条作为导读。
# 2.原则1：客户中心主义
企业应用架构必须建立在客户（Customer-Centric）的视角上，也就是说，企业应用架构的各个层次必须围绕着客户而不是供应商、开发人员或者管理者。这种“客户中心”观念强调客户的需求、喜好以及能力，是真正支持客户成功的关键。如果应用架构不考虑客户的真正需求，很可能会导致竞争对手侵犯客户利益甚至影响客户满意度的情况发生。通过将客户视为中心，企业应用架构可以更好地满足客户的需要，构建出好的用户体验。

# 原则2：简单性
企业应用架构不应该过分复杂化，否则将无法满足组织的需求。相反，企业应用架构的设计应该以最简单的形式呈现，让系统的开发、维护、运维和部署变得容易、快捷。简洁的代码实现通常会带来更加可靠、稳定的运行环境，缩短开发周期，降低风险。复杂的架构或设计往往会增加应用的复杂性，进一步引入新的错误，难以维护和扩展。

# 原才3：简单做好一件事情
企业应用架构的设计不应该陷入“完美设计”的陷阱中，也不会因为复杂的架构或设计而让项目延期，而是应该始终坚持“简单做好一件事情”这个原则。尽管某些功能或模块可能看起来很复杂或困难，但总的来说，企业应用架构的设计必须要简单到足够满足当前需求即可。

# 原则4：开放协作
企业应用架构设计需要面向多方面的了解，包括技术、业务、法律、政治、经济以及安全等各方面的因素。这些信息应该被充分利用，才能提供最佳的应用架构设计建议。通过开放协作，企业应用架构设计可以在不同部门之间分享信息，确保各方利益最大化。

# 原则5：统一的标准和框架
企业应用架构设计必须遵循一套统一的标准和框架，才能保证应用的一致性、正确性和完整性。统一的标准和框架能够促使不同部门的专业人士共同参与设计过程，提高效率和质量。因此，企业应用架构设计应当采用集体智慧，共同努力制定并落实标准和规范。

# 原则6：面向服务的体系结构（SOA）
SOA，即面向服务的体系结构，是一种架构设计模式，旨在通过服务的形式，把复杂的系统拆分成小型、可独立开发的模块，以提供给不同的应用使用。通过SOA架构设计，企业应用架构中的各个层次可以高度模块化，系统具有较高的灵活性和可伸缩性。SOA架构模式能够促进系统的模块化、组件化、分布式化以及服务化。

# 原则7：关注自身职责和使命
每一个企业都有自己的使命，为了达到这个使命，企业应用架构设计者必须去了解其业务领域、系统资源、组织架构、上下游关系等，找寻其所属的权力边界，确定自己的角色和责任，从而确定自己的职责范围，为自身的业务目标奠定基石。只有真正关注自身职责，且重视分配相应的资源，才能设计出有效的应用架构。
