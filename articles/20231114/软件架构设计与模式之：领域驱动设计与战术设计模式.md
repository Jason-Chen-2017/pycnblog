                 

# 1.背景介绍


软件架构设计（Software Architecture Design）是指在不违反业务规则、质量目标、组织要求等条件下，基于需求、功能特性、结构风格、框架约束及其他因素设计一个软件系统或者模块的结构，包括它的软硬件平台划分、组件选取和接口设计，进而完成开发，使其能达到预期的目标。目前最主流的软件架构设计方法是五层架构（分层架构、三层架构、四层架构、六层架构），各个层级之间通过不同的设计策略，可以实现信息系统或子系统的模块化、健壮性增强、可移植性、可扩展性和可管理性的提升。然而，传统的软件架构设计模式并不能完全解决复杂的业务场景下的架构难题，例如设计大型分布式系统中的微服务架构，设计业务系统中的事务处理、容错、数据一致性和业务流程管理等。因此，为了更好地应对复杂的业务场景，云计算行业、大数据技术、移动互联网时代带来的软件架构模式变革，以及服务端架构作为基础设施发展的趋势，我们需要了解更多的软件架构设计模式，其中战术模式(Tactical Pattern)占据了很重要的地位。
战术模式是软件架构设计中一种特殊类型的设计模式，它侧重于对整个软件系统架构的某些方面进行调整，以改善系统的性能、可用性、可伸缩性、安全性、可维护性等属性，以便能够更好地适应变化、提供高效、可靠的服务，比如用消息队列替代同步调用，用缓存减少数据库查询次数，使用异步处理方式提升系统吞吐率。战术模式并不是用来直接指导项目的架构设计的，而是在项目开始阶段就应该考虑将哪些战术模式纳入架构设计中。战术模式的实践还依赖于相应的工具、语言和框架支持，这些技术往往需要多年的积累才能成为习惯。因此，熟悉战术模式对于架构师来说至关重要。
# 2.核心概念与联系
战术模式是一类特殊的设计模式，它共有10种，它们大致按照类别和范围分别如下：

⑴ 创建型模式（Creational pattern）——用于处理对象创建机制的模式；

⑵ 情景模式（Contextual pattern）——用于描述交互场景的模式；

⑶ 结构型模式（Structural pattern）——用于描述类和对象的结构关系的模式；

⑷ 行为型模式（Behavioral pattern）——用于描述对象间的交互方式的模式；

⑸ 中介者模式（Mediator pattern）——用于简化对象之间的通信的模式；

⑹ 职责链模式（Chain of responsibility pattern）——用于向多个对象中转请求的模式；

⑺ 命令模式（Command pattern）——用于将一个请求封装成一个对象，从而使你可在不同时间指定、排队和执行该请求的模式；

⑻ 状态模式（State pattern）——用于允许对象自身的状态改变而自动触发动作的模式；

⑼ 观察者模式（Observer pattern）——用于建立对象间的一种一对多依赖关系，当一个对象改变状态时，所有依赖于它的对象都会得到通知并被自动更新的模式。
