                 

# 1.背景介绍

随着 internet 的普及和各种各样的互联互通应用程序的人口尖峰，建立可扩展、高性能、高可用性和弹性的软件系统变得越来越重要。负载均衡技术意味着以下两点：

1. 在集群内部进行很好的资源利用，在集群内部尝试自动化无压力的资源分配。选择“正在休眠”或“负载最低”或“最热门(或者负载)最高”或其他来分发工作以满足用户需求，而不是让超过集群中某个机器资源的激烈竞争
2. 在集群之间对流量实施充分的分发（可以为负载在各个集群之间的负载平衡）。这样就可以将整个集群对客户端的负载分配之间平均。

负载均衡技术得以建立在一些一些相关的概念之上：

1. *动态*的运行状况检查：负载均衡引擎应该在 clusters 健康的成分变化时快速发挥作用，比如，当一个成分宕机时，负载分配引擎应该能够自动将相关的流量在可用的机器中转移。
2. *可选择*的客户端：根据系统成分的“状态”路由流量应该可以做到客户端在集群的工作成分。可以是安装与代理的临时 IP / URL，也可能只是池成分的名字，或者标签。可能还有其他相关的信息，如“最少” / “最长” / “最快” / 平均响应时间等。
3. **架构：**负载均衡技术应该与其他组件可以统一的客户端应用程序和集群成分一起运行。其配置应该是动态的，而不是由硬编码的代码携带的。

这个领域的最新进展是，Sock Shop Demo，用于流量"负载"的虚拟集群的服务模拟器。可以在容器化中使用这种方法进建立一个集群，但是使用它时继续记住其原通信是单个接口的，但多个端口的侦听服务。

负载均衡的关键来源是：流量分发的规则，负载仪器有使用一些` Severity, Class, Likelyhood`的事件。

一个负载均衡实现应该考虑到：

1. 流量分发规则的定制性，如` host/url`，` client-ip`，` cookie-based`，` geolocation based etc`。
2. 健康检查，主动，被动，主动检查的定期读取或者TCP 资源，IP层 和 app 层一切。 corporation of metadata: port, path, protocol,URL path, server certificate, all arguments of ` timeo_connect`, timeout send, ` timeout_ucp`, and ` retry_dead_servers`
3. 如何在集群中撤销�?过期或停止的成分。
