Git是目前世界上最先进的分布式版本管理系统（DVCS），与其说它是一个系统，不如说它是一套完整的版本管理工具集，涵盖了诸如版本控制、分支管理、远程协作等功能。它的设计目标就是有效率、性能和简单性，这使得它成为许多开源项目的必备组件。
本文将带领大家使用Git进行版本控制，掌握日常工作中经常用到的各种Git命令，并应用到实际工作中去。
# 2.基本概念术语说明
## 2.1 Git基础知识
### 版本控制
版本控制是一种记录一个或多个文件内容变化，以便将来查阅特定版本修订情况的系统。主要目的是为了帮助开发者们更好地跟踪文件修改历史、管理复杂的代码bases以及进行团队协作。
### 分布式版本控制
分布式版本控制系统（Distributed Version Control System，简称 DVCS）是指在同一个仓库(repository)下，每个developer都可以independently工作而互不干扰，各自保存自己的change set，然后向服务器提交更新，由服务器根据收到的更新通知合并所有change sets，生成新的版本，从而保证仓库始终保持最新状态。
Git是最流行的分布式版本控制系统，但并不是唯一的分布式版本控制系统。还有其它分布式版本控制系统比如Mercurial、Bazaar等。其中Bazaar和Mercurial支持分布式开发，但由于历史原因（他们创建时期没有考虑分布式开发），因此不推荐使用。
### 工作区、暂存区和本地仓库
**工作区**：就是你在电脑里能看到的目录，当你第一次打开一个Git仓库的时候，这个目录就是工作区。
**暂存区**：英文叫stage或者index，是Git用来存储信息的中间区域。一般在你执行git add 命令后，改动的文件就会被复制到暂存区。暂存区里面存放着的是临时的修改，一旦你把暂存区的文件提交到HEAD，那么在commit时就已经将修改内容永久纳入历史了。
**本地仓库**：顾名思义，这是你的本地的Git仓库，当你执行完push命令之后，仓库的内容会同步到远程仓库。

### HEAD指针
**HEAD指针**：是一个指向当前版本（即最新提交版本）的游标。每次在克隆仓库时，Git自动创建一个master分支，此时master分支就是当前HEAD指针所指向的版本。如果HEAD指针指向其他分支，那就是切换分支了。
### 暂存区VS Commit
**暂存区**：暂存区是一种缓存区域，用于存放暂时还没准备好提交到版本库的文件修改内容。工作区中的文件并不会实时提交到暂存区中，只有运行add命令才会将文件添加到暂存区。此外，暂存区也可以看作是一个预览区域，你可以对暂存区中的内容进行编辑，修改后再提交。
**Commit**：提交，也称为快照，它是版本库的一个静态瞬间。提交完成后，它成为不可改变的历史记录，不可回滚。
### 远程仓库
远程仓库就是一个仓库，里面存放着某个项目的完整代码以及版本历史。
## 2.2 Git命令概述
Git命令是Git的基本操作单元，下面是一些基本的Git命令和它们的常用选项：
- `git init` 创建一个新的Git仓库。
- `git clone <url>` 从远程Git仓库克隆一个新的本地仓库。
- `git status` 查看当前仓库状态。
- `git diff` 比较暂存区和工作区的差异。
- `git add <file>` 将文件添加到暂存区。
- `git commit -m "message"` 提交暂存区的变更。
- `git log` 查看提交历史。
- `git branch` 查看分支。
- `git checkout <branch>` 切换分支。
- `git merge <branch>` 合并分支。
- `git push origin <branch>` 推送本地分支到远程仓库。
- `git pull origin <branch>` 拉取远程分支的更新。
