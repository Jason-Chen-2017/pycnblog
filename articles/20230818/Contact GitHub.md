# 1.简介
  

GitHub是面向开源及私有软件项目的代码托管平台，让您和世界分享您的代码，推动社区建设，并发现更多优质软件项目。近年来，越来越多的开发者开始使用GitHub进行项目管理、协作和开放源码贡献。在使用GitHub进行项目管理时，需要熟悉GitHub的基础知识、命令行操作、GitHub Flow工作流等，了解如何利用GitHub更好地管理代码库、制作优秀的Issue和PR描述，提升团队的协作效率，同时也可促进开源社区的蓬勃发展。

为了帮助开发者更好的掌握GitHub技能，我特意研究了GitHub的基础知识、功能特性和用法，编写了一系列技术文章，希望能够帮助到大家。本文为第1集，内容介绍GitHub的背景、功能特性和基础知识。
# 2.GitHub的背景
GitHub于2008年创建，是一个面向开源及私有软件项目的主机服务网站，目前已成为全球最大的同性交友网站，也是全球最大的开源社区。GitHub背后的公司称为GitHub Inc。2018年4月GitHub宣布完成B轮融资，估值超过17亿美元。截止2021年6月底，GitHub累计用户超过660万，日活跃用户超过5亿。根据GitHub Insights发布的数据显示，截至2021年6月，GitHub上总共有29.3亿个仓库（repository），其中包括开源和私有的，且每个仓库平均有78个 contributor。而且，GitHub的每周新增用户超过20万。

GitHub的主要功能特性如下：

1.版本控制系统 Git：GitHub拥有强大的版本控制系统Git，可以对代码历史进行记录、回滚、比对、合并等操作，同时可以与其他平台或工具整合，如Bitbucket、Azure DevOps等。

2.项目协作 Platform：GitHub提供完整的项目管理功能，包括基于Kanban、看板、冲刺计划等方式的项目管理工具，可轻松地管理复杂的项目流程和任务分配。同时，GitHub还提供了GitHub Projects和Wikis等额外功能，可用来跟踪工作进展、收集反馈信息等。

3.代码审查与提交 Review & Pull Request：GitHub提供一套完整的协作流程，包括通过代码审查与Pull Request提交的方式，提高代码质量、促进团队合作、促进开源社区发展。对于Pull Request来说，GitHub提供Code Review工具，开发人员可以在线讨论代码改动，保证提交的代码符合要求。

4.Issues & Notifications：GitHub具有完善的Issue管理功能，开发人员可以在Issues中提交错误报告、需求建议、任务分解等，同时GitHub还会将开发者关注的问题通过邮件通知给相应的负责人。

5.Pages服务 Pages：GitHub除了为代码提供版本控制、项目管理和代码审查功能外，还提供Pages服务，用于托管静态网页、生成文档、个人博客等，降低服务器维护成本。

6.Packages Registry：GitHub允许开发者上传自己的软件包，称为Packages Registry。开发者可以将自己开发的软件发布到这个仓库，广泛应用到各种开发场景。除此之外，GitHub还支持其他语言的开源项目，如Python、JavaScript等。

# 3.GitHub的基本概念术语说明
## 3.1 用户与组织
GitHub是一个面向开源及私有软件项目的主机服务网站，它不仅仅提供代码托管和版本控制服务，还可以通过各种功能扩展到其它的功能领域，例如项目管理、问题追踪、任务管理、Wiki、私信聊天、文档协作、CI/CD流水线自动化、ChatOps机器人等。这些扩展功能依赖于GitHub的一些基本概念和术语。
### 3.1.1 用户 User
用户指的是使用GitHub服务的个人或组织，他们可以创建和管理属于自己的Repositories（仓库）。每个用户都有一个个人页面，可以在该页面查看自己所创建的仓库、参与的活动、最近动态等。用户也可以设置通知选项、个人主页、个人资料、密码、SSH密钥、授权设备、组织成员身份等。
### 3.1.2 组织 Organization
组织是一个工作组，由多个用户组成，可共享资源、项目和协作流程。一个组织可以创建多个仓库、与多个用户协作、建立内部网络等。组织的管理员可以创建、删除仓库、修改成员权限、禁止外部成员加入等。组织还可与第三方平台集成，如Jira、Confluence、Trello等，实现项目管理、知识分享等功能。
## 3.2 Repositories（仓库） Repository
仓库又称为Repo，是GitHub的核心概念，用来存放文件的地方。每个仓库都有自己唯一的ID，包含了一组文件，可以使用Git版本控制系统进行管理。仓库中最重要的文件是README.md，它是对仓库内容的一个描述，通常包含关于仓库的介绍、使用方法、目录结构、FAQ、Bug报告规范等信息。另外，仓库还可以选择性地添加LICENSE、CONTRIBUTING.md、CODE_OF_CONDUCT.md、CHANGELOG.md、SUPPORT.md等文件。
## 3.3 Issues Issue
Issue是一个议题，一般表示某些功能或想法的请求或者问题。当某个功能或想法无法满足某个要求时，可以创建一个Issue，其他用户或开发者可以对其进行评论、投票、关闭、重启等操作。每个Issue都有一个编号、标题、描述、创建时间、标签、评论等属性。Issue通常用于对产品、功能或缺陷进行追踪，并且提供了一个讨论点。
## 3.4 PR Pull Request
PR即“拉取请求”，代表着用户向项目作者请求将自己在自己的分支上的更新合并到项目的指定分支(通常为master分支)中。当创建了一个PR之后，将请求通知给相关人员，让他们对PR做出评价和批准，只有被批准后才能将PR合并到目标分支。一个PR中可能包含很多修复、优化、新功能的代码。
## 3.5 Wiki
Wiki是GitHub提供的用于项目知识分享的功能。它是一个简单的HTML页面，可以随时编辑，任何人都可以访问。Wiki的内容包括项目文档、使用教程、FAQ、工作流程图、RFC模板等。项目所有成员都可以编辑Wiki页面，因此它非常适合作为项目管理工具。
## 3.6 Actions
Actions是GitHub提供的一项CI/CD功能，它允许用户定义一系列的操作，在每次仓库发生push、pull request时运行。用户可以自定义构建环境、执行测试、部署代码等，还可以集成各种第三方服务，如Slack通知、邮件提醒、ZenHub项目管理等。 Actions提供了一个高度定制化的开发流程，可促进协作和持续改进。
## 3.7 Branch
分支是用来存储开发版本的独立空间。在同一个仓库里，可以使用不同分支来进行不同的开发阶段。当一个分支达到稳定状态时，就可以合并到主干分支中。
## 3.8 Fork
Fork是在别人的项目上进行复制的操作，它的目的是为了参与他人的项目，但是由于复制的是别人的项目，所以并不是完全属于自己的。在GitHub的界面上，点击右上角的Fork按钮，可以将一个仓库Fork到自己的账户下。Fork之后，就相当于你在克隆别人的项目，拥有了原作者的权限，可以直接在本地进行开发、提交代码等。
