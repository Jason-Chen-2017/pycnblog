# 2.项目管理概述
## 2.1 什么是敏捷项目管理
敏捷项目管理（Agile Project Management）是一个过程方法学和工具集合，它鼓励迭代、循序渐进的方法和工具，以适应快速变化的需求、短期内完成项目目标的方式，通过反馈、调整和重视小步快跑的方式保证项目成功。敏捷项目管理的理念和实践由硝烟中的竞争力组合而成，包括用户驱动、团队动态和自组织能力等，可以有效支持复杂的软件开发生命周期。因此，敏捷项目管理已经成为一种软件开发流程，也是一种现代软件工程的重要组成部分。
## 2.2 敏捷项目管理的特征
### 2.2.1 适应性
敏捷项目管理能够满足项目环境的变化，适应快速发展的市场需求。敏捷方法把项目分解成更小的可交付成果，在短期内完成项目的目标，以反馈、调整和重视小步快跑的原则保证项目的成功。这种敏捷性质使得敏捷项目管理能够及早响应业务的变化，避免过度依赖于计划和设计，具有弹性和韧性，能够适应不断变化的需求，最终满足用户的需求。
### 2.2.2 个体和小组精神
敏捷项目管理鼓励一切以人为中心，充分尊重个人的贡献和价值，而不是只强调项目的成功，更注重每个人的成长与发展。通过个体之间的协作和创造力，完成更多的任务，产生更好的结果。敏捷项目管理还鼓励创新和尝试不同的想法，促进团队成员的探索和反馈，让团队有更多的空间进行批判性思维，推动创新。
### 2.2.3 跨越部门边界
敏捷项目管理能够跨越不同部门之间的界限，消除部门之间的隔阂，通过将知识和技能进行整合，提高软件开发效率。敏捷方法支持软件开发各个阶段的多方参与，降低沟通成本，并使产品的交付满足客户的需求。通过这种方式，项目经理可以向其他部门传达他们对软件开发领域的理解和经验，增强项目的透明度和可见性，促进知识共享。
### 2.2.4 清晰定义和可衡量的目标
敏捷项目管理有清晰的目标和任务定义，产品开发生命周期的每一个环节都有明确的预期目标和评估标准。这样的目标有利于持续跟踪进度，确保质量，并减少不必要的紧张。通过这些目标，项目经理和开发人员能够清楚地了解自己目前的工作情况，并根据实际情况调整工作节奏和分配资源。
## 2.3 敏捷项目管理的理论基础
敏捷项目管理理论包括敏捷宣言、敏捷价值观、敏捷过程模型、敏捷计划约束条件、敏捷变化管理、敏捷软件开发实践和敏捷转型论。下面我们分别简要回顾一下这些理论。
### 2.3.1 敏捷宣言
2001年，罗伯特·迪凯爾（Robert Decker）提出了敏捷宣言，它包含以下五条指导原则：
1. 个体和互动胜过过程和 tools
2. 工作的软件胜过详尽的文档
3. 客户 collaboration 胜过合同谈判
4. 响应变化 quick 的反应胜过 plan ahead
5. 对人而言开放性胜过隐蔽性
### 2.3.2 敏捷价值观
敏捷价值观认为，“软件行业的竞争环境如此之艰苦，我们不能再以往的简单做法”。为了应对这个挑战，“敏捷”一词被提出。敏捷包括三层含义：1) 不断反省，不断学习；2) 以用户为中心，以终端为中心；3) 快速而高效。它强调敏捷方法中对软件生命周期各个环节的高度关注。通过小规模的敏捷实践，项目经理和开发人员就能在较短时间内构建高质量的软件产品。
### 2.3.3 敏捷过程模型
敏捷过程模型是描述敏捷项目管理过程的一种框架。它包括固定阶段、迭代模型、验收模型、软件包模型、角色与责任模型、风险管理模型、会议模型等。迭代模型通常用于确定短期迭代的大小，让项目经理及开发人员有机会从用户反馈中获得信息，识别新需求。验收模型用于测试软件是否满足用户的要求，验证功能正确性和性能，并识别潜在的风险或bug。
### 2.3.4 敏捷计划约束条件
敏捷计划约束条件是敏捷项目管理的重要组成部分，它描述了如何安排项目开发、测试和部署的活动，以及管理项目过程中产生的各种异常情况。最常用的约束条件是冲刺计划、迭代计划、增量计划、问题管理、工作重点管理等。
### 2.3.5 敏捷变化管理
敏捷变化管理是敏捷项目管理的关键原则之一，它强调软件开发过程中应该不断适应变化，并且应该快速响应变化，确保软件持续更新、维护。敏捷变化管理通常包含三个方面：1) 及早识别变更，并适时做出反应；2) 采用敏捷开发模式，快速迭代软件版本；3) 在瀑布式开发中引入敏捷的迭代过程，使用适当的工作流减少回归风险。
### 2.3.6 敏捷软件开发实践
敏捷软件开发实践是使用敏捷方法构建软件时的具体原则和方法。如敏捷宣言中的工作优先级排序，面对压力时保持冷静，独立思考，降低沟通成本等。敏捷软件开发实践包含测试驱动开发（TDD），BDD，持续集成（CI），代码 reviews，结对编程，敏捷组织文化等。
### 2.3.7 敏捷转型论
敏捷转型论认为，“只有认识到敏捷的核心价值，以及如何应用该方法，才能成功转变为敏捷开发的正道”。它强调学习，在没有系统的组织、文化和工具支撑下，靠个人力量也无法成功转变为敏捷开发。因此，转型论鼓励组织和个人逐步转变，每一个人的贡献都是有意义的，因为只有所有的人才可以形成更强大的敏捷开发团队。
# 3.敏捷项目管理策略
## 3.1 Kano模型
Kano模型是一个传统的敏捷项目管理模型，可以用来检验敏捷项目管理的有效性。它的四个步骤是：1) 招募需求，2) 分配资源，3) 执行计划，4) 检查进度。Kano模型源自美国学者Jerome Kano提出的概念模型，它是一种描述项目流程的方法，可以用直观的语言描绘出项目开发的过程。Kano模型可以帮助判断公司正在使用的项目管理策略是不是符合敏捷方法，也可以判断开发团队是否掌握敏捷项目管理的技能。
### 3.1.1 招募需求
Kano模型的第一步是招募需求。招募需求时需要考虑多个因素，比如市场竞争，外部压力等。团队在开始前一般会围绕着用户的真正需求，收集真实的数据和反馈，形成用户故事，从而搭建需求模型。
### 3.1.2 分配资源
Kano模型的第二步是分配资源。这里需要考虑的是开发技能水平，开发环境和团队构成等。团队可以分成多个专门的小组，有经验的开发人员可以参与其中，帮助新人入门。
### 3.1.3 执行计划
Kano模型的第三步是执行计划。Kano模型将项目流程分成不同的阶段，每个阶段都有一个固定的目标和任务。项目经理和开发人员在完成需求调研后，制定计划，计划的长度取决于需求的复杂程度和剩余时间。计划一般由两部分组成：关键任务和非关键任务。关键任务是在接下来的几个迭代中完成的，在当前迭代结束之前不会发生变化；非关键任务需要在几个月甚至更久之后才完成，并可能被拆分成多个小任务。
### 3.1.4 检查进度
Kano模型的第四步是检查进度。检查进度时，项目经理和开发人员都会花费大量的时间来监控计划的完成情况。检查的内容可以包括研发进度、测试进度、资源利用率、风险识别等。如果发现任何差距，项目经理和开发人员就可以根据计划表对问题进行纠偏。
## 3.2 如何处理项目管理的鸿沟
在传统项目管理中，很容易出现“计划-承诺-执行”的鸿沟。计划阶段会制订项目的大方向，承诺阶段会设定关键任务的进度和关键里程碑，但实际上很多任务却并没有按计划执行。而在敏捷项目管理中，计划阶段则会制订迭代的优先顺序，承诺阶段则会制订每个迭代要完成的工作，而且任务执行也会受到真实数据和反馈的影响。“计划-承诺-执行”的鸿沟则不存在了，因为迭代是短暂的，只执行了一部分工作后，就会进入下一轮迭代。因此，敏捷项目管理的承诺会比计划更加细化，更能反映实际情况。
## 3.3 建立和维护项目管理关系
建立项目管理关系是敏捷项目管理的一个重要方面。项目经理应该建立起清晰的责任分配制度，明白开发和测试任务之间的关系，做好团队激励和激情培养工作。同时，项目经理还应该持续关注市场需求、技术进步和外部环境变化，密切跟踪这些变化并制订相应的变更计划。项目管理关系还可以帮助项目经理更好地组织团队、监督进度、防止风险、和客户保持良好的沟通。
## 3.4 技术债务危机
敏捷项目管理虽然能够应对需求变更、快速响应变化和弹性资源消耗，但仍然存在着一些技术债务的问题。例如，技术债务一词表示由于项目未能实现目标而导致项目延期或者失败。项目经理应当尽早发现技术债务并采取补救措施，以提高软件的生命周期质量。
## 3.5 运营项目管理
运营项目管理是敏捷项目管理的一个重要组成部分。运营项目管理包括商业模式、品牌管理、客户关系管理、销售、市场营销、财务管理等。这类工作涉及到市场营销、金融投资、销售等多种业务，是促进公司快速扩张的关键环节。项目经理应当深刻理解公司战略和商业模式，构建适合公司业务发展的项目管理机制，掌握有效运营项目管理的方法。
