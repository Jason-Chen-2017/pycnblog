# 1.简介
         
　　敏捷开发（Agile Development）是一个非常热门的话题，它的出现主要源于国际敏捷开发联盟（IDAL）等非营利组织对敏捷宣言的推行。与传统瀑布开发方法相比，敏捷开发更加关注项目过程中的细节和及时反馈，要求尽快交付可用的软件产品。敏捷开发倡导开发人员应该主动关注需求变更、持续不断改进软件、拥抱变化，而不应该被过多的流程和工具束缚住手脚。
         　　但是，作为一个强调迭代的方式，敏捷开发也容易受到一些人的误解，认为它是一个反模式或者一种错误的方式，即把敏捷开发的核心精神完全颠倒了过来。虽然敏捷开发在实践中取得了成功，但它也存在着诸如极度缺乏客户参与、僵硬的管理结构和极端的技术专横等问题。因此，本文将从一些基本的认识和常识出发，试图回答以下两个重要的问题：敏捷开发究竟为什么是一个反模式？又该如何正确使用敏捷开发？
         　　 
         # 2.敏捷开发背后的哲学思想
         　　“敏捷”这个词最早由罗伯特·德鲁克提出来，他在其著作《敏捷宣言》中将敏捷开发定义为：
           “每个软件项目都应当以用户故事为核心，通过自我组织和自我协调的方法完成。个体和互动始终是首要的，而工具只是支持他们运转的辅助设施。” 
           在这个定义中，敏捷开发没有给出一个具体的时间表或工期，而是在强调通过小规模快速迭代的方式来完成开发工作。另外，它也很重视协同和反馈，要求所有的团队成员参与其中并能够充分了解整个过程。
         　　当今敏捷开发已经成为企业和个人开发者之间的共识。很多公司如IBM、亚马逊、微软、Facebook等都采用了敏捷开发方法进行软件开发，并且有专门的敏捷开发咨询公司帮助企业落地敏捷开发。但是，对于许多初创型公司来说，这种方式会带来不少的困难，因为这些公司往往缺乏足够的经验和资源来管理和执行敏捷开发。
         　　此外，随着技术的革新和快速变化，技术人员需要迅速适应新的技术框架和编程语言，因此敏捷开发也会面临新的挑战。比如，敏捷开发缺乏技术指导和学习指导，因此会导致很多初创型公司望而却步，错失良机；另一方面，敏捷开发方法的依赖性和复杂性也可能会限制初创型公司的发展空间，使其陷入越走越窄的恶性循环。
         # 3.敏捷开发的误区
         　　“反模式”一词最早出现在《The Mythical Man-Month》一书中，作者认为“项目管理最危险之处就在于企图用‘完美’的方法做大项目”。例如，说某个组织遵循敏捷开发方法可以得到高效的结果，实际上这一说法只不过是一种炒作。敏捷开发有自己的理念、方法论、标准，并且严格执行，而不是像传统项目管理方法一样，漫无目的地、盲目地干一些杂乱无章的活。因此，若真的以敏捷开发来指导项目管理，那么一定不能盲目地复制其流程和方法，而应该具体分析问题、寻找根本原因，并围绕这个原因制定针对性的措施。
         　　敏捷开发也有着自己的规则、实践、模式，比如sprint计划、迭代、用户故事、冲刺、红板报、看板等等。这套流程和方法不是一蹴而就的，而是随着时间的推移不断演化和优化的。因此，如果某些实践在实践中出现偏差，甚至恰恰相反，那么也只能说明这个实践可能还不够成熟。因此，正确使用敏捷开发需要在实践中不断总结经验教训、优化流程、引入新技术，并且站在业务、产品、技术、管理四个维度全面考虑。
