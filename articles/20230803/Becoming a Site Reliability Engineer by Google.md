# 1.简介
         
2021年，是近几年里“黑五”的高峰期。这个时候，传言马斯克和SpaceX等公司正在崛起，上市股票也在快速飙涨。当下，全球各大互联网公司纷纷布局海外市场，而国内一线互联网公司却缺乏相应的团队、资源来应对这波“黑五”。2021年的最后一个月，随着新冠肺炎疫情在全球扩散，许多企业已经陆续宣布解除运营或进入灾难恢复阶段。越来越多的行业将面临新的机遇，站在风口浪尖上的互联网企业却无力应对突如其来的变局。
          
          在这样的背景下，Google推出了Google Cloud Platform（GCP）这一云计算服务平台，让开发者、系统管理员以及产品经理都能够轻松部署、管理和扩展自己的应用程序。通过利用GCP，企业能够更加关注应用的性能、可靠性和可用性。而作为云计算领域最有经验的一批人之一，肯尼斯·罗宾逊（<NAME>）、托马斯·诺埃尔·索拉诺（Tom Sterno）、蒂姆·库克（Tim Cook）、史蒂夫·麦卡沃伊（Steve McKay）和雅各布·莱特利（Yoshifumi Littiri）四位工程师不约而同地选择了站在这个风口的行列，组建了Google Site Reliability Engineering Team (SRE)进行网站可靠性工程。
          
          本文就是关于站点可靠性工程的研究报告。我们的目标是以Google SRE团队成员的视角，从网站可靠性工程的前世今生、现状到未来发展方向，全面阐述并展开探讨。希望大家能从中收获更多的启发和思路。
         # 2.基本概念术语说明
         ##  2.1 Google Cloud Platform
         GCP 是 Google 提供的基于云的计算服务平台。它提供强大的计算、存储、网络功能，可以帮助用户快速构建和部署应用程序。例如，开发人员可以使用 GCP 的大量工具、API 和服务来创建、测试、部署和运行应用，包括虚拟机、容器、数据库、分析、机器学习、图形、IoT 等。
         
         Google SRE Team 从事的主要工作就是构建和维护 GCP 中的系统，确保这些系统能够在任何时间、任意位置正常运行，且具备高可用性和可靠性。因此，了解 GCP 的相关概念及技术细节至关重要。例如，GCP 由多个组件构成，如 Compute Engine、Cloud SQL、Kubernetes Engine、Storage、Networking 等。每个组件均拥有自己独特的能力和特性，开发人员需要对这些组件有一个全局认识，才能充分发挥 GCP 的优势。同时，理解各个组件之间的交互关系、依赖关系，以及各自的应用场景也是十分关键的。
         
         此外，要充分掌握 GCP 中提供的各种服务，例如 BigQuery、Cloud Dataflow、Cloud Functions、Cloud Monitoring、Cloud Storage、Pub/Sub 等，它们之间的区别和联系，如何结合起来一起使用，才是真正掌握 GCP 核心技能的关键。
         
         GCP 还提供了许多丰富的工具和 API，可用于自动化部署和配置资源。开发人员可以利用这些工具、API 来自动创建、删除、缩放服务器，实现应用的横向扩展和数据分析等。
         
         此外，除了基础设施层面的知识，Google SRE Team 还需熟练掌握 GCP 的管控和运维知识，包括安全性、访问控制、监控、日志、事件响应、备份策略、迁移方案、灾难恢复、故障排查、容量规划等方面。
         
        ## 2.2 Site Reliability Engineering （SRE）
         SRE 是 Google 一支工程团队，负责构建和维护 GCP 中的系统，确保这些系统能够在任何时间、任意位置正常运行，且具备高可用性和可靠性。其使命是通过使用可靠的自动化手段，来提升 Google 内部和客户的体验。
         
         SRE 团队通过以下方式来提升 Google SRE 技能：
         1. 建立知识体系：每天阅读技术论文，参与 Google 内部的技术分享会议，并且定期参加内部的 SRE Summits。
         2. 深入学习：在 Google 的不同业务部门之间共享 SRE 技术和经验，与其他团队一起进行技术沉淀和分享。
         3. 拓展视野：在业界和学术圈探索新的领域，与世界级的工程师保持联系。
         
         当然，SRE 不仅仅是一群工程师，它还包括行政、法律、产品、设计、销售、营销、财务等各个角色的专业人员。这些角色在 Google 中扮演着重要的角色，是 SRE 团队提升自身能力的重要途径。
         
         此外，SRE 需要坚持不懈地实践。作为工程师，SRE 需要面对日复一日复杂的任务，甚至可能成为项目延期、交付压力增加、质量保证不足的绊脚石。但是，SRE 不应该被单纯的技术所局限，他还需要学会倾听用户需求，向产品经理和用户推荐解决方案。如果不能做到这一点，那么 SRE 就很容易陷入误区，陷入自我封闭、狭隘自我改善的境地。
         
        ## 2.3 Website Reliability Engineering
         网站可靠性工程是 SRE 团队针对网站健壮性的一种工程方法，主要目的是减少网站出现故障的可能性，降低网站故障带来的影响范围。其包含三个子工程：
          
          1. 可用性工程：负责处理用户访问网站时的网络连接、硬件故障、系统错误等异常情况。
          2. 性能工程：重点关注网站的响应速度、加载速度、页面大小等指标，确保网站的正常使用。
          3. 容量规划和弹性工程：分析当前网站的容量瓶颈，制定相应的容量调整计划，最大限度地提升网站的稳定性。
           
         通过将以上三个子工程紧密联系起来，SRE 可以为网站的健康和快速发展提供有力的支撑。
         
         ### 2.3.1 可用性工程
         可用性工程通过对网站的可用性进行持续观察、评估、修正，以达到网站尽可能长时间保持正常运行的目标。
         
         可用性工程在 SRE 的工作流程中占据着重要地位，但又不应该忽略它的价值。由于不可避免的网络故障、硬件故障、第三方服务故障等原因，网站的可用性往往是整个系统的单点故障。为了减少网站的不可用时间，SRE 需要识别、预防、修复这些单点故障，确保网站始终处于正常状态。
         
         一般情况下，可用性工程可以分为两步：
         
         1. 漏洞检测：检查网站的各项服务是否满足可用性要求。
         2. 测试和修复：对网站进行性能测试，找出所有潜在的问题，并根据测试结果对网站进行修正。
         
         在漏洞检测过程中，SRE 需要检视网站服务的健康状况，以及这些服务之间的关系。比如，某个后端服务是否存在超时、失败率过高的问题；某个前端服务是否因请求量过大导致负载过重；某个外部服务是否存在雪崩效应等。SRE 还需要分析日志文件，找出异常的网络流量或访问行为，帮助定位问题根源。
         
         当发现漏洞时，SRE 会启动测试和修复过程。首先，SRE 将尝试重现出网站的某些异常情况，以确认是否是由网站本身的问题引发的。然后，SRE 会使用诊断工具对问题进行诊断，提取有效信息，进一步分析和理解问题。
         
         有时，SRE 可能会遇到某些难以复现的网络故障、硬件故障等问题。这种情况可能是由第三方服务引起的，比如 CDN 服务或者浏览器插件问题。在这种情况下，SRE 只能依赖于第三方服务的发布者来提供支持。另外，SRE 还需要采取积极的应急预案，立即升级第三方服务的版本号，或者在发生故障时切换到备用服务。
         
         可用性工程还可以为用户提供服务，并帮助用户解决网站访问中的困难。比如，网站可以通过友好的错误提示，指导用户解决登录或支付相关的问题。此外，网站也可以为用户提供咨询热线，帮助用户寻求技术支持。
         
         ### 2.3.2 性能工程
         性能工程关注网站的响应速度、加载速度、页面大小等指标。它可以帮助 SRE 识别慢速访问、卡顿、页面加载缓慢、加载过多数据的现象，从而对网站的结构、数据库查询、缓存设置、静态文件压缩等方面进行优化。
         
         性能工程的目标是提升网站的响应速度、降低网站的负载、减少用户等待时间，从而让用户获得更好的用户体验。由于网站服务的特性不同，其优化过程也各不相同。比如，对于大型购物网站，其响应时间需要在秒级内，而对于小型的个人博客站点，其响应时间则可以降低到毫秒级。
         
         性能工程可以采用 APM（Application Performance Management）监控工具来收集网站的性能数据。APM 工具可以记录网站各项性能指标，包括响应时间、加载时间、页面大小、HTTP 请求次数等。SRE 使用这些数据来判断网站的运行状态，帮助识别性能瓶颈，并采取相应的优化措施。
         
         以 Google Analytics 为代表的网站访问统计工具可以帮助 SRE 跟踪网站的访问数据。SRE 可以从数据中发现访问的地理分布、设备类型、搜索词、访问浏览器等，从而更好地分析网站的访问模式，找到访问的热点区域和地点。
         
         如果网站经常出现掉帧、卡顿、白屏等情况，SRE 可以通过分析浏览器日志、服务器日志、截图、网络抓包、分析线程 dump 文件等手段，找出问题所在，进行定位和分析。
         
         ### 2.3.3 容量规划和弹性工程
         容量规划和弹性工程是衡量网站可用性和性能的方法。它以网站的总体容量为基础，确定增减网站资源的数量和配置，以便网站承受更大的访问量和负载。
         
         容量规划和弹性工程有助于 SRE 平衡性能与可用性之间的权衡。SRE 需要根据网站的实际容量和负载，识别其容量瓶颈，制订相应的容量调整计划，最大限度地提升网站的稳定性和可用性。
         
         比如，Google 的搜索引擎使用 Google BigTable 存储索引数据，如果该数据集溢出，就会造成搜索响应时间变慢。SRE 可以通过扩大 BigTable 集群的磁盘空间，或购买更快的 SSD 存储硬盘来增加 BigTable 的容量。
         
         更多地说，容量规划和弹性工程还可以帮助 SRE 评估云平台的容量，验证网站的容量调整是否有效果，并排除任何潜在的容量限制。
         
         在容量规划和弹性工程中，SRE 还应当注意管理好资源分配。SRE 应当避免将所有资源都分配给一个或几个团队，以免资源过度分散，导致团队之间资源竞争。相反，SRE 可以将资源分配给多个团队，并合理调配资源，使得每个团队的工作负荷得到充分的执行。
         
         还有，容量规划和弹性工程还应当考虑网站服务的依赖关系。比如，一个网站依赖于另一个服务，如果这两个服务出现故障，那么网站的可用性就无法保证。因此，SRE 应当为网站服务建立健壮的依赖链条，确保服务间的通信畅通。
         
         最后，容量规划和弹性工程还需了解数据库容量的限制。由于一些网站会使用关系型数据库，因此 SRE 需要考虑数据库的容量限制。如果数据库的容量耗尽，网站的运行也会受到影响。SRE 需要定期对数据库进行维护，并及时清理不必要的数据。
         
         ## 2.4 Kubernete
         Kubernetes 是 Google 提出的开源容器编排引擎，具有高度的扩展性和自动化特性。它允许用户轻松地创建、部署、扩展和管理容器化的应用。Kubernetes 能够管理跨多个主机和公共云环境的容器化应用，并提供有效的资源利用率。
         
         SRE 团队的工作就是对 Kubernetes 集群进行维护和优化。其中，Kubernetes 的管理和运维工作占据了最高的优先级。Kubernetes 的架构和核心组件都需要经过深入的理解，才能更好地掌握 Kubernetes 的运维知识。
         
         ### 2.4.1 Kubernetes 控制器
         Kubernetes 是一个基于容器的集群管理系统，其核心组件包括 Kubelet、kube-proxy、etcd 和 API Server 等。这些组件都是 Kubernetes 控制器的运行单元。
         
         每个控制器都负责管理集群中特定功能的实体，包括节点、pod、副本集、存储卷和服务。因此，掌握 Kubernetes 控制器的工作原理和工作流程，对于管理 Kubernetes 集群、维护服务的高可用性、提升集群的资源利用率等方面，都有重要的意义。
         
         Kubernetes 控制器通常以循环的方式运行，按照一定顺序周期性地读取集群中的事件、资源对象，并依据其定义的规则进行处理。控制器的主要职责有：
         
         1. 创建、更新、删除资源：这些控制器负责创建、更新、删除集群中的资源对象，例如，Pod、Deployment、Service、Ingress 等。
         2. 调度 pod：调度器负责决定 Pod 在哪台节点运行。调度器的主要作用是保证应用在集群中的均匀分布，并将应用的高可用性要求与资源利用率目标进行兼顾。
         3. 扩容、缩容集群：这些控制器负责扩容和缩容集群，例如，扩容集群中应用的数量，或缩容死亡的节点。
         4. 监控集群：集群的监控控制器负责监控集群中的资源消耗、节点健康状况、应用健康状况、集群资源使用率等。
         
         Kubernetes 的控制器机制可以保证集群的稳定运行，并有效地管理集群的资源。因此，掌握 Kubernetes 控制器的工作原理，对于 Kubernetes 的管理和运维都有着十分重要的意义。
         
         ### 2.4.2 Kubernetes 基础设施
         Kubernetes 的基础设施包括集群的网络、存储、计算等资源。Kubernetes 提供了丰富的接口和插件，允许用户自定义资源。自定义资源可以用来描述各种类型的集群资源，例如，有状态应用的存储卷声明、无状态应用的服务声明等。
         
         用户可以编写配置文件或模板文件，用以定义 Kubernetes 对象。配置文件或模板文件可以帮助用户快速创建和管理 Kubernetes 对象。同时，用户也可以使用命令行工具、API 或 UI 来管理集群资源。
         
         Kubernetes 提供了多种存储接口，包括本地存储（NFS、iSCSI）、远程存储（Ceph、GlusterFS）、云存储（AWS、Azure、GCE）等。用户可以利用这些存储接口，为 Pod 配置持久化存储。
         
         Kubernetes 支持多种计算资源，包括 CPU、GPU、FPGA、TPU、Heterogeneous Computing（异构计算）等。用户可以为 Pod 配置不同的计算资源请求。同时，Kubernetes 支持弹性伸缩（Horizontal Scaling），允许用户按需调整集群中 Pod 的数量。
         
         Kubernetes 还支持集群扩展插件，可以根据用户的需求安装和卸载插件。扩展插件可以为集群添加新的功能，例如，CRI（Container Runtime Interface）插件可以实现容器的运行时接口，CSI（Container Storage Interface）插件可以提供存储功能。
         
         在 Kubernetes 中，集群节点的管理也属于 Kubernetes 基础设施的一部分。用户可以通过 kubectl 命令行工具或 API 来管理集群节点，包括查看节点信息、监控节点状态、管理节点资源等。
         
         Kubernetes 基础设施的管理对于 SRE 角色来说非常重要。掌握 Kubernetes 集群节点的管理技巧，可以让 SRE 最大限度地发挥集群的资源优势，为组织提供高效、稳定的基础设施服务。
         
         ### 2.4.3 Kubernetes 应用
         Kubernetes 应用指的是部署到 Kubernetes 集群中的应用，例如，微服务、容器化应用、Batch Job、Serverless Function 等。
         
         Kubernetes 提供了丰富的应用管理工具，包括 Deployment、StatefulSet、DaemonSet、Job、CronJob 等。用户可以通过 YAML 文件或 Helm Charts 来定义应用的元数据，并使用 kubectl 命令行工具或 API 来管理应用。
         
         Kubernetes 应用的管理也需要 SRE 进行相应的操作。SRE 可以为 Kubernetes 集群中的应用定义相应的资源请求、限制、亲和性等规则，以及应用的健康检查和滚动升级策略。
         
         Kubernetes 应用的管理还涉及到应用性能优化。SRE 可以使用 Prometheus、Grafana、EFK Stack（ElasticSearch、Fluentd、Kibana）等工具来收集、分析和监控 Kubernetes 集群中的应用性能数据。
         
         ### 2.4.4 集群自愈
         当集群中的某个资源不足或出现问题时，集群自愈机制可以自动为集群补齐资源、重新调度 Pod、恢复服务。
         
         Kubernetes 自愈机制有两种，即垃圾回收机制和弹性伸缩机制。
         
         垃圾回收机制负责删除不需要的资源对象，例如，删除完成的 Job 和已终止的 Pod。
         
         弹性伸缩机制能够根据集群中资源的使用情况，动态增加或减少集群节点的数量。当集群中资源不足时，弹性伸缩机制会触发集群的自动扩容操作，来为集群提供更多的资源。
         
         集群自愈机制对于维护 Kubernetes 集群及其运行的应用非常重要。SRE 可以根据集群自愈的触发条件，采取相应的措施，以确保集群的高可用性、稳定性和资源利用率。
         
         ### 2.4.5 Kubernetes RBAC（Role-Based Access Control）
         Kubernetes 提供了基于角色的访问控制（RBAC）机制，允许用户根据角色来划分权限。RBAC 机制可以帮助用户管理 Kubernetes 集群的访问控制，并限制敏感资源的访问权限。
         
         Kubernetes 中的角色包括集群管理员（Cluster Admin）、命名空间管理员（Namespace Admin）、项目管理员（Project Admin）、集群视图者（Cluster Viewer）、命名空间视图者（Namespace Viewer）、项目视图者（Project Viewer）等。
         
         SRE 可以通过配置 RBAC 规则，对 Kubernetes 集群中的资源、命名空间和项目进行细粒度的权限控制。SRE 可以使用 kubeconfig 文件、IAM（Identity and Access Management，身份与访问管理）、OIDC（OpenID Connect，开放 ID 认证协议）等方式进行用户鉴权。
         
         因此，掌握 Kubernetes RBAC 的工作原理和工作流程，对于 SRE 角色来说至关重要。
         
         ## 2.5 Promethues + Grafana + EFK Stack
         上述 Kubernetes 相关技术虽然具有重要的实践意义，但是还是远远不够。只有正确掌握了 Kubernetes 的管理和运维技能，才能更好地使用 Kubernetes 实现云原生应用的管理。
         
         为了更好地管理 Kubernetes 集群和应用，SRE 需要了解 Kubernetes 集群中的服务和应用。Prometheus+Grafana+EFK Stack（Elasticsearch、Filebeat、Kibana）是一种很好的管理工具组合。
         
         Prometheus 是由 CNCF（Cloud Native Computing Foundation，云原生计算基金会）主办的一款开源系统监测软件。它可以从目标服务拉取各种指标数据，包括 CPU、内存、网络流量、磁盘 I/O、系统负载等。Prometheus 可以对数据进行时间序列聚合、过滤、存储和展示，形成直观的图表。
         
         Grafana 是一款开源的可视化和仪表板工具。它可以将 Prometheus 拉取的数据呈现成丰富的图表和图形，并提供丰富的可视化分析功能。Grafana 还可以与 Prometheus 的数据源进行集成，从而实现数据透视表的构建。
         
         Elasticsearch 是一款开源的搜索和分析引擎。它可以对 Prometheus 生成的指标数据进行索引、存储和检索。EKF Stack（Elasticsearch、Filebeat、Kibana）是一套完整的日志管理工具集合。其中，Elasticsearch 负责存储和分析日志数据；Filebeat 负责采集日志数据；Kibana 负责呈现日志分析结果和数据可视化。
         
         因此，Prometheus+Grafana+EKF Stack 是一个完美的日志管理工具组合，能有效地管理 Kubernetes 集群中的服务和应用。SRE 可以使用 Prometheus 和 Grafana 来收集和分析 Kubernetes 集群中的服务和应用的性能数据，并通过 EFK Stack 来管理和呈现日志数据。
         
         ## 2.6 Docker
         在 Kubernetes 管理和应用中，Docker 也扮演着至关重要的角色。Docker 帮助用户打包、构建、共享和运行容器化的应用。
         
         SRE 需要对 Docker 镜像和容器进行管理和优化。Docker 提供了一系列的工具，可以用来构建、上传、下载镜像，以及管理容器运行时环境。
         
         SRE 可以使用镜像签名机制来确保镜像的完整性。镜像签名机制可以验证镜像是否来自信任的源头，并帮助避免镜像被篡改或伪装。同时，镜像仓库还可以实现镜像分发和共享功能。
         
         SRE 可以使用容器运行时（Container Runtime）对容器进行隔离和限制。不同的容器运行时可以提供不同的资源限制和安全性保证。Kubernetes 支持 Docker、containerd 和 CRI-o 等多种容器运行时。
         
         SRE 可以配置 Kubernetes 中的PodSecurityContext，为容器指定特定的用户和组，以便容器可以访问主机文件系统和网络。同时，SRE 可以使用Volume（卷）功能，为容器提供持久化存储。
         
         最后，SRE 可以使用 DaemonSet 和 NodeAffinity 等功能来管理节点，为集群提供统一的资源配额和调度策略。
         
         ## 2.7 Prometheus Operator
         Kubernetes 的管理和应用是不可替代的，但是 Prometheus 的集群管理仍然是一个难点。很多时候，用户需要自己手动管理 Prometheus 集群，而 Prometheus Operator 提供了一个解决方案。
         
         Prometheus Operator 是 Kubernetes 官方提供的用于部署、管理和维护 Prometheus 集群的 operator。它可以帮助用户快速、方便地创建和管理 Prometheus 集群。
         
         Prometheus Operator 使用 Custom Resource Definition（CRD）来定义 Prometheus 对象。用户可以在 Kubernetes 中创建 CRD 对象，来声明 Prometheus 集群所需的配置参数。Operator 根据这些参数，通过调用 Prometheus 官方提供的 API 来创建 Prometheus 集群。


         Prometheus Operator 的优点主要有：
         
         1. 统一管理 Prometheus 集群：用户只需要定义一次 Prometheus 配置参数，Operator 就可以自动创建和管理整个 Prometheus 集群。
         2. 简化 Prometheus 集群的操作：Operator 封装了 Prometheus 操作的复杂逻辑，用户只需要关注 Prometheus 集群的管理和运维即可。
         3. 自动化的扩容和缩容：当 Prometheus 集群出现资源不足或扩容时，Operator 会自动扩容或缩容集群中的节点。
         
         因此，掌握 Prometheus Operator 的工作原理和工作流程，对于 SRE 角色来说也至关重要。

         
# 3 站点可靠性工程的研究工作

## 3.1 Google SRE团队的研发背景

2010年9月，谷歌正式进入全球化的时代。当时，Google 曾经在美国的某个研究机构担任科学家，他的研究兴趣正好与互联网的发展方向一致。谷歌最早的搜索引擎 SerpRank 最初源自于科学家们的研究，它基于用户点击行为及相关链接的行为习惯，通过分析用户的搜索历史和网页结构，计算出网页的相关性。

2012年，谷歌投入大量资金开始布局海外市场。由于移动互联网的兴起，谷歌决定开始在亚太地区推广搜索引擎，提供更快的服务。为了适应海外市场的用户，Google 引入了亚马逊（Amazon）和搜狗（Sogou）等巨头，将其旗舰产品进行改良和优化，提升了搜索引擎的用户体验。

2013年，谷歌决定将搜索服务完全转向搜索广告，打算在全球范围内吸引流量，并通过各种手段提升广告收益。虽然谷歌在去年几乎所有的搜索广告业务都以失败告终，但是在这之前，它一直秉持着在线广告、实时竞价、搜索引擎、移动应用联合运营的理念，这也促使其在互联网和广告业界占有一席之地。

2014年初，谷歌面临着搜索引擎市场的最大挑战——垄断。因为谷歌一直以来都是为大众着想，所以它一直坚持自家搜索引擎的霸主地位。Google 的搜索引擎曾经成为谷歌电商的核心服务。但随着搜索引擎技术的不断革新，以及用户需求的变化，谷歌逐渐放弃了自家搜索引擎的霸主地位，转而投入资源进行创新，开发出更加先进的搜索引擎产品。

2016年，谷歌又一次陷入了垄断的局面——在电商市场快速崛起的背景下，搜索引擎垄断行业显得尤为艰难。在这之后，谷歌决定重新审视搜索引擎市场，重新定义搜索引擎的定义，并引入新的市场竞争力——云计算。

2017年，谷歌宣布将其搜索引擎产品整体转型为云端产品，并开始了云端搜索引擎的研发工作。在谷歌的精心设计下，云端搜索引擎可以实时搜索用户的搜索请求，并快速响应。在节省成本的同时，云端搜索引擎还能够提供更好的用户体验，提升搜索服务的效果。

2018年，Google SRE团队在Google X的提案下成立，旨在建立一个能够为所有产品提供可靠性和可扩展性的服务，通过运维软件、平台和基础设施的自动化实现这一目标。该团队由工程师、产品经理、营销专家和财务专家组成。他们的目标是在Google服务的所有层面上实现可靠性，包括基础设施、软件、产品、应用程序、API等。

 ## 3.3 关键词

- Availability：网站的可用性，通常是指网站服务的时间与间隔。
- Latency：网站的响应速度，即网站返回请求的时间，通常是指从用户输入查询到页面呈现给用户的时间。
- Fault tolerance：网站服务的容错性，通常是指服务在遇到任何问题时，依然可以继续运行的能力。
- Scalability：网站服务的可扩展性，通常是指网站的处理能力在业务增长时可以自动扩张或缩减。
- Resiliency：网站服务的韧性，通常是指网站在面对突发事件（如洪水、火灾、地震等）时仍然可以保持正常运行的能力。
- Security：网站的安全性，通常是指网站的内容、用户信息等在传输、存储和处理过程中被保护不被泄露的能力。
- Configuration management：配置管理，是管理计算机系统配置的过程，涉及到安装、配置、更新和变更软件、硬件、网络设备、应用软件等的过程。
- Automation：自动化，是指使用脚本、自动化工具和流程，将繁琐重复性的工作自动化，简化并加快生产、测试和部署的过程。
- Service Mesh：服务网格，是一种用于治理和监控微服务的体系结构，其特征是利用微服务架构的服务网格，以轻量级代理来劫持网络通信，实现安全、可靠的服务间通信。
