# 2.基本概念术语说明
首先，我们需要明确一下一些基本的概念和术语。

1.代码提交(Commit)：代码提交即将本地修改的代码推送到远程仓库(Repository)。每个提交都有一个唯一的标识符号commit id。每一次提交都会自动生成一个新的提交id，并且会在整个开发生命周期中持续记录所有的提交信息。

2.代码冲突(Conflict):代码冲突是指两个或者多个开发人员同时对同一份代码进行编辑而导致代码出现错误。在发生代码冲突时，需要手动解决冲突才能完成提交。

3.代码风格(Code Style):代码风格是一套编码规范，它规定了一种编程语言中语法结构和表达方式的方针。

4.单元测试(Unit Test):单元测试是验证代码正确性的过程，它可以测试模块间的交互逻辑和边界情况，以及各种输入输出条件。单元测试一般是在开发过程中进行，并不是所有的代码都需要编写单元测试。

## 3.1 代码审查的意义
代码审查的主要目的是为了发现代码中的错误，检查代码的一致性、健壮性、可读性和完整性，还可以有效地改善代码质量。代码审查不仅可以有效地发现问题，而且可以促进团队合作，提升软件质量，减少缺陷，增加软件的可维护性、复用性和扩展性。下面是一张图展示了代码审查的工作流程:

![code-review](https://cdn.jsdelivr.net/gh/geekhall/pic@master/img/20211013_233945.png)

1.代码提交: 项目成员将编写完毕的代码进行提交，提交的信息应该详细准确，包括提交人的姓名、邮箱地址、提交日期、提交内容、提交影响范围等。

2.代码评审: 对代码进行逐行的审查，查看是否存在以下问题：
    - 语法错误
    - 命名规范违反
    - 代码可读性差
    - 测试用例漏补
    - 功能实现缺失
    - 文件包含过多或过小的文件

3.提交结果反馈: 根据审查结果给出建议，对提交者进行代码修改，并再次提交。

4.重新评审: 如果提交者修改后提交的代码仍然存在问题，则重复上述过程。如果提交者提交的代码没有问题，则结束审查。

## 3.2 代码审查的几个要素
- 查错：代码审查的第一步就是对代码进行查错，这一步就像是在寻找瑕疵一样，必须要系统、全面、仔细地检查代码，避免引入错误。
- 提醒：代码审查过程中还会遇到各种各样的问题，审查者应当把注意力集中到代码质量相关的地方，必要的时候才做指导性的建议。
- 学习：代码审查最重要的技能就是学习能力，审查者必须对代码的实现原理有深刻的理解，能够快速理解代码的设计思想和实现方法。
- 效率：代码审查的效率直接影响到整个软件开发流程的顺利程度，因此必须掌握代码审查的方法和工具，提高效率，降低管理成本。
- 专注：审查者不能盲目地进行“打死”审查，应该深入分析问题，只做有助于问题解决和提升代码质量的事情。

## 3.3 代码审查的流程
1. 安装Git：代码审查依赖于Git，所以需要安装Git。
2. 配置Git：配置用户信息，设置默认编辑器和合并工具。
3. 创建分支：在Git里创建新分支。
4. 检查代码规范：检查提交的代码是否符合代码风格要求，例如空白字符、缩进、变量名、函数名、注释等。
5. 执行单元测试：执行单元测试，确保提交的代码满足单元测试的条件。
6. 提交代码：提交代码，如果发现提交的代码有问题，可以进行二次提交。
7. 删除分支：删除分支。
