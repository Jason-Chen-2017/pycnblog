# 1.背景介绍


## Prompt Engineering 是什么？
Prompt Engineering (PE) 是微软亚洲研究院 (MSRA) 和中国交通科技大学 (Tsinghua University) 在今年推出的新的领域创新项目。该项目旨在通过 AI 技术自动生成高质量的测试用例，并通过人工智障能辅助进行测试改进，提升测试效率、可靠性和质量。

从细节到整体，PE 有以下几个主要价值:

1. 更高质量的测试用例生成: PE 机器学习算法可以自动生成具有更高质量的测试用例，通过对代码的分析、数据采集、数据标注等多方面手段，它可以生成具有更强的业务逻辑和场景感知能力，帮助开发人员快速编写高质量的测试用例。
2. 测试效率的提升: 通过自动生成测试用例，PE 可以降低测试工程师手动编写测试用例的时间，缩短测试周期，使测试效率得到明显提升。
3. 可靠性的保证: 通过加速测试用例的迭代更新，PE 可以确保测试用例在每个阶段的验证结果都是可信的。
4. 质量改进的意愿: PE 的测试辅助工具可以让测试工程师更好的理解测试用例中的错误原因，基于这个原因改善测试用例，实现测试用例质量的不断优化，改善软件产品的可靠性和用户体验。
5. 对所有人员的价值贡献: 除了 AI 生成的测试用例，PE 提供了相关的工具链支持和流程，帮助测试工程师、开发人员和质量保证工程师共同参与测试工作，形成对整个过程的参与和协作。

PE 涉及多个领域，比如自动生成测试用例、测试辅助工具、测试流程、知识库构建等，它的特点也是很吸引人的，但同时也存在一些局限性。例如:

1. 算法耗时长: 从训练数据到生成测试用例需要一定时间，这会影响开发人员的生产力。
2. 模型准确度不高: 目前的算法模型还不能完全覆盖业务需求，同时还存在很多模型误判的情况。
3. 生成的测试用例只能是少量简单案例: 虽然算法可以自动生成测试用例，但它们仍然需要手工审核和编辑。
4. 缺乏系统性方法论: 由于当前算法模型生成的测试用例仍处于初级阶段，没有统一的方法论指导测试用例的设计。

本文着重讨论的是第一个问题——如何解决可测试性问题。
# 2.核心概念与联系
## 可测试性
定义：一个系统或模块是否容易被正确地测试是一个非常重要的问题。如果一个系统或模块不易于测试，那么它的质量将无法得到保证。因此，为系统或模块建立测试用例的过程就是为了消除或减轻其中的不可测因素。

可测试性一般由以下三个因素决定：

1. 功能性可测试性：如果一个系统或模块的所有功能都可以通过简单的输入输出组合来进行测试，那么它就具备了功能性可测试性。对于一些复杂的系统和模块来说，比如操作系统、网络协议栈、数据库系统等，功能性可测试性往往较差。
2. 可理解性：如果一个系统或模块内部结构较复杂，或者涉及到的算法或机制比较多，那么其可理解性就会较差。
3. 可部署性：如果一个系统或模块要部署到其他环境中进行测试，那么它的可部署性就至关重要。不同平台之间的兼容性、第三方依赖库的版本控制、配置管理等都会造成可部署性的困难。

总而言之，可测试性是一个系统或模块的所有三个维度的综合表现。越完美的系统，其可测试性越好。只有做到了功能性可测试性，才能保证系统的质量，否则只会导致系统开发过程中的混乱和风险。

## 本文所属主题——提示词工程(Prompt engineering)
Prompt Engineering 是微软亚洲研究院（MSRA）和中国交通科技大学（Tsinghua University）于 2021 年底推出的新领域创新项目。该项目旨在通过 AI 技术自动生成高质量的测试用例，并通过人工智障能辅助进行测试改进，提升测试效率、可靠性和质量。

Prompt Engineering 的关键特征包括：

1. 以“提示”的方式向开发者提供可执行的测试用例，而不是以常规的模板形式；
2. 提供可理解的测试用例，能够反映系统的实际运行状态；
3. 使用机器学习技术生成高度优化的测试用例，能有效缩短测试用例编写时间，提升测试效率；
4. 通过人工智障能辅助改善测试用例，以便更好地满足需求，提升测试的可靠性和可维护性。

Prompt Engineering 作为一个整体框架，把系统中各个模块的测试需求、实现方案、组件间通信、环境依赖关系等全都考虑在内。这种系统级的视角可以促进测试用例的设计，达到精细化、自动化和系统化的效果。
