# 1.简介
         
## 1.背景介绍
自动化测试(Automation Testing)是指将已验证的软件功能或流程按照规定的规范、流程和测试方案来运行测试用例，以发现并报告其中的错误、缺陷等，提升软件质量的过程。软件测试的主要目标就是在尽可能少的成本下发现软件中的“瑕疵”，提高软件质量。
而对于软件测试来说，传统上主要有两种方法：黑盒测试（Black Box Testing）和白盒测试（White Box Testing）。
## 2.基本概念术语说明
1. Test Case: 测试用例，也称用例描述或者需求文档，是对一个特定功能或者用例的描述，用来反映程序员对这个功能的理解程度。主要包括测试目的、输入条件、预期输出以及测试环境等。

2. Requirements: 需求文档，是测试人员和开发人员在项目中明确而精准地定义所需软件特性、功能及性能要求的有效文件。它主要用于协助测试团队定义产品范围以及确定需求的优先级，在敏捷开发模型中，通常由业务分析师、产品经理、用户体验设计师或管理者编写。

3. Unit Tests:单元测试，又叫模块测试，是对一个软件组件进行正确性检验的测试工作，它只关注该模块是否能够正常工作且具有良好的边界性。单元测试通常覆盖模块的接口、输入、输出和异常处理。

4. Integration Tests:集成测试，是将不同模块、子系统、类库、函数等组装成一个完整的功能系统，然后进行测试。它通过将各种部件之间相互作用、协同工作的情况进行模拟，评估各个部件的结合是否符合设计。

5. System Tests:系统测试，是对整个软件系统的全面测试。系统测试涉及到多个模块、子系统、类库、函数、数据库、应用服务器、Web服务器等。系统测试要验证整个系统的各个方面的功能、兼容性、鲁棒性、可用性等。

6. Functional Tests:功能测试，是指按照用户的使用场景测试软件的某些功能是否符合预期。功能测试是指按照用户的真实需求或场景测试某个软件功能是否能够实现，并对比实际结果和预期结果，评价其准确性、完整性、可靠性和可移植性。

7. Smoke Tests:冒烟测试，是一种简单的测试，用来检查一个产品的基本功能是否可以正常运作。通常冒烟测试由产品的开发人员、测试人员、测试团队或其他相关人员主动运行，以检测软件的基础功能是否能正常运行。

8. Regression Tests:回归测试，是指开发新功能后，验证之前已存在的功能是否仍然正常工作。回归测试保证了新功能不会影响到现有的功能。
9. User Acceptance Tests (UAT): 用户验收测试，是指测试人员向最终用户展示最新版本的软件产品，让他们对产品功能、性能、界面效果等进行验收。之后再进行详细测试，确认软件的缺陷和效率方面是否达到了要求。
10. Accessibility Tests: 可访问性测试，是一种验证软件是否满足残障人士、弱视人士、网络不稳定因素、分辨率低、颜色弱、语言等特殊用户群的需要的测试。可访问性测试的目的是更好地满足这些用户群的需求。
11. Performance Tests: 性能测试，是衡量软件性能的重要工具，目的是判断软件在某些特定条件下的运行速度、内存占用、CPU负载、磁盘读写等。性能测试还可以监控软件在不同时段的平均响应时间、吞吐量等。
12. Stress Tests: 压力测试，是在高强度或长时间运行情况下对软件系统的稳定性、健壮性和安全性进行测试，以发现软件中潜在的故障或问题。
13. Fuzzing Tests: 模糊测试，是指随机输入，目的是找出软件代码中隐藏的漏洞和逻辑错误。模糊测试的目的是寻找各种边界条件、输入组合、无效输入、恶意攻击等问题。
14. Security Tests: 安全测试，是指识别、防范和缓解计算机安全事件发生所带来的危害，从而保障数据的安全、隐私和完整性。
15. Compatibility Tests:兼容性测试，是对软件在不同的操作系统、平台、硬件配置等环境下运行的兼容性进行测试。目的是检查软件是否能正常运行，即所需的外部依赖项是否都已经安装成功。
16. Interoperability Tests: 互操作性测试，是指验证不同软件之间的数据交换和通信协议是否符合标准协议。它可以帮助发现、隔离和解决不同系统间的差异性问题。
17. Accessibility Tests: 可访问性测试，是指验证软件是否具备足够的可访问性，以便残障人士、弱视人士、网络不稳定因素、分辨率低、颜色弱、语言等特殊用户群正常使用。可访问性测试的目的是更好地满足这些用户群的需求。
18. Cultural Fit Tests:文化匹配测试，是指验证软件产品是否符合企业文化的风格和语言习惯，避免产品出现语言和文化符号的歧义。
