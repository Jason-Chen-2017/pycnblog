# 1.背景介绍


持续集成和持续部署（CI/CD）已经成为IT界最流行的两个技术。它们能够帮助软件团队更快、更频繁地将新代码集成到主干上并发布出去。随着云计算的普及，越来越多的组织都转向了基于云平台的自动化运维模式。这些模式可以轻松地实现应用部署、环境配置和服务更新等任务。但是，持续交付和持续部署只是实现自动化的第一步。在现代企业里，还有许多其他领域也需要跟上其脚步，比如构建、测试、监控等，才能让公司内部的生产力不断提升。因此，持续交付和持续部署之外，还需要进一步探索如何打造可靠且高效的敏捷开发流程。
敏捷开发是一个反应快速变化的需求的开发方法论。它鼓励客户和项目管理人员协同工作，把产品开发过程拆分成小的、可测试的增量块，并且以迭代的方式进行交付。它的核心价值观是通过简化流程和优化开发流程，减少重复性的工作，提升软件开发人员的工作效率。
持续交付和持续部署只是实现敏捷开发的一部分。如何结合起来更好地实现敏捷开发，是持续交付和持续部署的重点所在。

# 2.核心概念与联系
## 2.1 什么是持续交付与持续部署？
持续交付与持续部署是一种新的软件开发方法，可以看做是持续集成（Continuous Integration，简称CI）和持续部署（Continuous Deployment，简称CD）相互促进的产物。

持续集成意味着团队成员在每一次代码提交后都能立即获得反馈，使得代码始终处于可用状态。持续集成工具可以帮助我们自动执行构建、静态分析和单元测试等过程，确保代码质量始终处于较好的状态。

持续部署则意味着在每次代码合并到主干或者某个预生产环境中之后，自动触发部署流程，将最新版的软件部署到线上。对于一些复杂的部署流程，可以通过配置工具自动化部署。

两者的组合可以实现自动化构建、测试、集成、部署等过程，这可以极大的提升开发效率，缩短开发周期，降低部署风险。

## 2.2 CI/CD与敏捷开发之间的关系
持续集成/持续部署流程与敏捷开发之间并没有必然的联系。他们的目的不同，各有侧重。

持续集成和持续部署主要关注的是自动化流程，而敏捷开发主要关注业务目标和快速响应变更。两者之间并不是孤立存在的。

例如，持续集成与自动化测试一样，是为了减少反复编写测试用例，提升开发效率。但是，持续集成所提供的能力比传统测试更多，可以对应用中的各种细节进行测试，还能检测代码间依赖关系。所以，持续集成不是一无是处，它与敏捷开发的结合可以帮助研发人员完成更复杂的自动化测试。

另一方面，持续部署也给企业带来巨大的投入，不过通常情况下，它并不会占据核心竞争力。因为，企业往往还需要考虑各种自动化运维、监控和备份方案。而敏捷开发关注的是业务目标和快速响应变更，一般只需很少的人力参与即可。所以，持续部署可能是敏捷开发的一个补充。
