# 1.简介
  

## 什么是Git？
Git是一个开源的分布式版本控制系统，被设计用来快速有效地处理小到非常大的项目版本管理。它与其它版本控制系统相比，最显著的特征就是它的速度。它一般只需很少的网络带宽，就可以克隆、推送、搜索任意一个历史提交版本，甚至可以从不同的分支中比较不同时期的变化。Git有许多优点，诸如：
- 技术先进性：Git采用了松耦合的方式，使得其在各个子系统之间交流变得简单直观。通过Git命令，用户可以轻松实现各项工作，例如分支合并、标签管理等。因此，很多公司都将Git作为团队开发工具。
- 分布式特性：在Git的官方网站上有一个很棒的动画图表展示了分布式版本控制系统Git在不同场景下的分布式特性。可以看出，由于Git支持分布式特性，所以其仓库能够处理上百万级的文件，并且在本地存储仅需要几个G的空间。这对于一些海量文件的版本控制来说是非常友好的。
- 速度快：虽然Git本身性能很强悍，但还是有些慢。不过，Git在创建新分支和解决冲突时速度却不俗。另外，Git还包括高效的内部优化机制，能够使操作更加快速和可靠。
- 安全性：Git拥有成熟的安全措施，可以防止数据泄露、恶意攻击等。同时，Git也提供了各种认证方式，提升了安全性。

Git具有这些优点，因此越来越多的人开始用它来进行版本控制。而现在，越来越多的人正在学习并使用Git。那么，什么是Git，它有哪些重要概念和术语呢？我们需要了解一下。
# 2.基本概念术语
## 概念定义
### 工作区(Working Directory)：就是你在电脑里能看到的目录，这里面存放着实际文件。

### 索引区(Index/Stage Area)：英文缩写为"IDX"，暂且称之为缓存区吧。基本上，所有修改过的文件都被放在这儿等待提交。

### 版本库(Repository/Repo)：工作区有一个隐藏目录".git",这个目录含有指向索引区和其他信息的数据库文件。

### 暂存区（或叫做"Stash"）：英文缩写为 "STASH"，用于临时保存和恢复你的工作进度。主要目的是为了让你方便地切换分支或者修复bug，而不是像commit一样永久保存。有的时候你在修复了一个bug，结果发现有新的更紧急的问题需要解决，但是你不想因为旧的bug影响到新的修复。这时候就需要stash一下，然后继续修复新的问题。

### 分支（Branch）：每个版本库都可以拥有多个分支。当你要开发一个新功能或者搭建某种环境时，你会基于某个分支新建一个分支。这样的话，你可以自由地在两个分支之间开发，最后再把它们合并。比如，你正在开发一个新功能A，但是目前在master分支上，但突然接到产品经理的电话说，要建立一个新环境B，于是你便checkout到另一个分支develop创建一个新分支。

### HEAD指针：是每一个Git仓库都会有的指针，指向当前所处的分支和提交。HEAD随着指向的变化而自动更新。

### 提交（Commit）：在版本控制系统中，提交是指对项目的一个整体快照进行永久性记录。每次提交都会生成一个哈希值（SHA-1），用于标识项目的状态。当然，你也可以手动指定提交信息来帮助自己梳理项目记录。

### 远程仓库（Remote Repository）：当你从远程仓库克隆时，会得到一个新的Git仓库。这个仓库有一个远程地址（remote URL）和相应的读写权限，你可以通过该地址对远程仓库执行推送和拉取操作。

### 拉取（Pull）：从远程仓库获取最新版本到本地仓库，也可简称为抓取。

### 提交（Push）：将本地仓库中的改动上传到远程仓库，也可简称为推送。

### 标签（Tag）：标签是一个简单的文本标记，用于给重要的版本打上标签。通常情况下，标签都是轻量级的，除了包含标签名称外，不会额外增加其他任何信息。标签通常是通过快照来创建的。

### 对象（Object）：是Git的核心数据结构。Git把数据存储为一组对象，每个对象表示一个文件的内容、一组版本化的信息或者一个树结构（即子目录）。可以使用Git hash-object命令手动创建对象。

### 引用（Ref）：是指向Git对象的指针。HEAD是一个特别的引用，它指向当前所在的提交。通常情况下，引用会指向 commits 或 heads ，但是也可以指向其他类型的对象。

### 命令行接口（Command Line Interface）：在终端下输入命令来操作Git。CLI是一种用户友好的命令界面，使得用户不需要记住繁琐的参数或长长的命令，只需一条条敲命令即可完成各种操作。

### 撤销（Revert）：撤销操作会将指定文件的更改撤销到之前的版本，让文件回到最近一次提交时的样子。
