# 1.简介
         
　　软件设计是一个复杂而繁琐的过程，也是一门艺术。为了更好的实现软件的功能、性能、扩展性等特性，工程师们一直努力寻找方法来简化软件的设计过程。其中一个有效的方法就是设计模式。
         ## 为什么要学习架构模式？
         　　学习架构模式，首先需要明白“模式”到底意味着什么。传统意义上的“模式”是一个已经存在过的解决方案或者模板。它不仅仅局限于某个特定的领域或技术，而且还可以应用于不同的场景。软件开发中，我们经常会遇到一些重复出现的问题，这时候就可以借助设计模式来减少开发时间和降低错误率。但同时，架构模式同样也具有抽象、封装、多态等特点，有利于提高软件的可维护性和可扩展性。
         ## 架构模式与设计模式的区别
         　　1. “模式”一词的用法不同。“模式”一词通常指的是已有的解决方案，如英国画家G._Moore所创造的四次抽奖游戏，即“模范牌、中奖幸运号码、拾光不存、连胜奖”。此时，“模式”一词只是对已有的方法或模板的一种称呼。
         　　　　2. 而在软件开发领域，“模式”一词表示的则是一种通用的、可重用的解决方案，一般来说指某种类型或主题的解决方案，例如：建筑风格、数据库设计模式、系统架构模式、单元测试框架、MVC模式、J2EE模式等。
         　　　　3. 在学习架构模式的时候，一定要注意分清两者之间的界限。
         　　2. 作用范围不同。软件架构模式通常只涉及到软件系统的总体设计和架构设计，设计出来的架构应当能够反映出整个软件系统的结构、层级和职责划分。因此，它往往适合于系统建设初期，可以帮助架构师和技术负责人理顺软件设计的方向和制定开发计划。而设计模式则不仅仅局限于软件开发领域，也可以用于其他各个领域，如图形用户界面设计、人机交互设计等。
         　　3. 抽象程度不同。软件架构模式中的模式是抽象的，抽象程度比设计模式更高，更具代表性和普适性。比如说，软件架构模式中的代理模式、策略模式、状态模式都属于结构型模式；而设计模式中的单例模式、模板方法模式、适配器模式等都属于创建型模式。
         　　4. 方法论不同。软件架构模式注重系统结构、层级和职责划分，而设计模式更侧重于具体的实现细节和技巧。软件架构模式强调以系统为中心，把握全局，将系统模块化、信息隐藏起来，并遵循“开闭”原则；而设计模式强调抽象、封装、多态，并且还提供了相应的实现方法。
         　　5. 关注点不同。软件架构模式主要关注系统的整体架构和设计方式，以便于管理和维护；而设计模式则着重于实现的精确性和效率，包括设计原则、编程规范和编码标准等方面。因此，它们往往是相辅相成的。
         　　综上，若从实际角度来看，学习软件架构模式还是很有必要的。因为只有掌握了模式的思想、方法、技巧，才能更好地理解和实践面向对象软件设计的各种模式，并且能够在软件开发中游刃有余地进行架构设计。
