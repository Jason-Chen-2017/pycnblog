# 1.简介
  


现在的开发者已经没有时间和精力去手动测试他们的代码了，不管是功能测试还是单元测试都需要耗费大量的人力、时间和金钱。现在很多开发者担心自动化测试是否真正起到作用，担心编写自动化测试用例的时间成本过高等。但是，如果忽视自动化测试带来的好处的话，那就太可悲了。在这个问题上，开发者也有自己的看法，比如认为自动化测试不足以保证项目质量、引入自动化测试的风险比较高、自动化测试无法覆盖所有的情况、自动化测试不利于维护等等。

那么，究竟什么原因导致开发者不重视自动化测试呢？

## 为什么要重视自动化测试

开发人员总体来说并不是很重视自动化测试，原因主要有以下几点：

1. **忽略价值所在**。
2. **被现实打败**。
3. **盲目追求**。
4. **不情愿承担责任**。
5. **失去控制感**。

**第一点，忽略价值所在。**从产品或项目的生命周期角度出发，测试并不是只有开发阶段才有必要，测试也应该作为整个过程的一部分进行，它能够帮我们发现很多bugs，提升产品质量，而不仅仅是开发人员在做完功能后再去手动测试，更有可能是在开发过程中就已经考虑到了测试工作。而且，自动化测试还可以结合DevOps流程，更加高效地对代码进行检测和部署。

**第二点，被现实打败。**作为一个IT从业者，我们经常会面临各种各样的问题，比如时间管理、工作压力、知识产权保护等等。尽管自动化测试能够减少一些重复性工作，但是它仍然不能完全消除人为因素造成的错误，所以仍然需要手工测试来验证系统的正确性和健壮性。而且，自动化测试还不能完全替代人的工作，因为仍然存在一些任务需要靠人来执行，比如负责部署的运维工程师、需求分析、设计文档等等。

**第三点，盲目追求。**一直以来，自动化测试都是被一些技术人员所青睐的，许多公司都会聘请这些人去开发自动化测试工具。不过，随着年轻人的增长，越来越多的技术人员开始自我怀疑，觉得自己的技术水平并不足以成为一名高级测试工程师，甚至还有些不屑一顾的意思。于是，一些新的技术也慢慢浮现出来，如DevOps、云计算、容器技术等等。对于这些新技术的认识，一些老旧的技术人员却把它忽略了。

**第四点，不情愿承担责任。**对于任何的系统，都应当有充分的测试来确保其正常运行，但测试并非免费的午餐，一份好的测试用例往往需要花费更多的时间、资源、金钱。无论如何，开发者对测试的依赖程度不会低于其他部门，这是不可避免的。因此，如果开发者不重视测试，则只能通过“闭门造车”的方式来完成一些功能，这显然是不合理的。

**第五点，失去控制感。**由于个人能力的局限性，开发者往往更倾向于采用自己的想法来实现一些功能。这种态度可能会导致自动化测试被视为“另类”，甚至导致它被看作是一种“异端”。这也是为什么很多时候自动化测试被认为是一种空中楼阁，永远也不会被重视。反观传统的开发流程，早期的开发人员花更多的精力在代码的编写上，后期才渐渐转入测试环节。反观现在的自动化测试流程，测试人员扮演了一个重要角色，开发人员需要更加重视测试工作。

## 为什么要建立自动化测试

目前，全球有超过6亿的移动应用用户，平均每天有超过3万个应用安装上线。这样庞大的市场规模带来了巨大的流量压力，为了应付这样的压力，许多公司都在逐步推行自动化测试来提升应用的质量。一般情况下，自动化测试是作为持续集成的一部分来进行的，可以及时发现缺陷，缩短上线时间。自动化测试有助于提高开发速度、降低风险、提升质量。

自动化测试的目的主要有两个方面：

- 提升代码质量。自动化测试可以帮助检查代码中的bug，让开发人员及早解决潜在的问题，减少bug出现的概率；
- 测试过程自动化。自动化测试可以将测试过程自动化，不需要开发人员参与，使测试更加准确可靠，从而提高测试效率。

基于以上两点，自动化测试成为非常重要的一种软件测试方法，它的推广迅速，也得到了开发者的广泛关注。目前，许多企业已经开始利用自动化测试来提升软件质量。

## 自动化测试为什么失败？

自动化测试的不断普及虽然给了开发者更多的方便，但同时也暴露出了一个严重的问题——自动化测试并非万能灵药。尤其是在复杂的应用程序中，自动化测试也经常会遇到一些问题。

首先，人机交互的复杂性。自动化测试涉及到与浏览器、手机、电脑等设备的交互，并涉及到用户界面上的各种元素，这就要求自动化测试人员具有良好的理解能力和技巧，能够准确地定位页面上的元素。但是，自动化测试人员往往又是技术宅，他们在解决问题的时候往往还会遇到一系列困难，比如网络延迟、屏幕分辨率差异等。因此，自动化测试人员可能会花费大量时间来调试脚本，直到脚本能够正常运行。

其次，系统测试的复杂性。自动化测试往往依赖于系统的一些外部服务或者组件，如数据库、消息队列等，而这些组件的可用性、性能和稳定性又是需要高度关注的。因此，自动化测试的结果可能受到这些外部依赖的影响，并导致测试结果的不确定性。

最后，业务规则和场景测试的复杂性。自动化测试往往只测试开发者手动测试的某些特定的功能点，但往往忽略了程序中隐藏的逻辑错误。比如，假设一个购物网站的下单功能正常工作，然后加入一项促销活动，就可以预期买家点击按钮下单之后获得不同商品。但是，如果某个促销活动并没有真正成功，那么下单操作就会产生误导性的行为，可能导致购物金额异常。因此，自动化测试需要更加全面的测试策略，包括业务规则、场景测试等。

## 自动化测试该怎么做才能有效果？

这里有一些建议供大家参考：

1. **覆盖范围广**。自动化测试的覆盖范围不能只是测试应用的功能，还需要包括一些边界条件和异常情况的测试。即便是刚开始做自动化测试的初学者，也不要贸然盲目地跳进自动化测试的坑里。一定要充分了解应用的特性、业务规则和场景，从多种角度来测试应用，探索更多的可能性。

2. **关注细节**。自动化测试不仅要覆盖应用的所有功能，还要保证测试的细节程度。比如，每个接口请求的参数设置是否符合规范、响应时间、日志输出是否符合要求等，都需要关注。在缺乏经验的情况下，最好找一位熟悉相关领域的专业技术人员来指导和协助。

3. **不拘泥于标准**。自动化测试只要能够达到目的，不管多么复杂的应用，都可以得到较好的效果。但是，要注意不要陷入“标准就是唯一的答案”的思维模式，要根据实际情况制定适合自己团队的测试方案。

4. **测试框架选择**。自动化测试工具繁多，且不同语言和平台也有不同的自动化测试框架。开发人员需要根据项目的实际情况选择适合的自动化测试框架，并遵循相关的编程规范和规则。

5. **小心排查**。自动化测试往往会引起整个测试过程的暂停，因此，测试人员也需要掌握快速定位问题的能力。除了用眼睛察觉外，也可以尝试打印日志信息、分析报错堆栈等手段来判断问题的根源。
