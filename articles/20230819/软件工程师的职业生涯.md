# 2.背景介绍
作为一名优秀的软件工程师，首先需要搞清楚自己身处的这个行业环境，以及这个行业对社会和个人的意义。下面简单介绍下软件工程师的相关背景。

1. 行业发展变化

软件工程师这一行业近几年在全球范围内迅速发展，在过去十多年里，全球各地都出现了不同规模的软件公司，如苹果、微软、谷歌、Facebook、Twitter、Uber等。这些公司都拥有着庞大的开发者队伍，其中很多软件工程师都是他们的顶尖员工。

当然，随着科技的不断发展，软件工程师也在日益成为全球最紧缺的人才之一。世界卫生组织发布的最新调查显示，在2020年，全球约有90%的劳动力被认为是缺乏技能的。而在这个行业，有着十分重要的工作岗位需求，如系统工程师、软件工程师、前端工程师、后端工程师、运维工程师、数据库工程师等。

2. 市场需求

对于企业来说，软件工程师的角色具有重要的作用。目前，软件工程师已经成为“数字化转型”过程中的关键环节。软件工程师负责开发、维护、优化软件产品，从事研发、测试、部署、运营等全生命周期管理工作。他们能够解决复杂且紧急的需求，并制定相应的策略、计划、工具和流程，确保软件的质量与安全性。因此，企业越来越依赖于软件工程师的专业才能，能够帮助企业有效整合资源、降低成本，提高效率、减少风险。

此外，软件工程师还扮演着“知识转移”的角色，帮助企业迁移到云计算领域，促使更多的软件工程师加入到软件开发的队伍中。

3. 人员分布

在软件工程师的职业路径中，有三种类型的人员分布情况。第一种是体系内职位，即公司内部的技术资源池。体系内职位主要由开发人员、测试人员、架构师、数据库管理员、项目经理、产品经理、项目助理等。第二种是外部培训机构的技术人员，如亚马逊AWS架构师认证、微软MVA认证、Udacity认证、Coursera认证等。第三种是实习生，这些实习生通常来自于IT外包、创业公司、教育机构、政府部门等。

4. 学习习惯和职业路径

软件工程师的学习习惯可以归纳为专注、训练、反馈三个要素。首先，软件工程师必须高度专注于学习，不得懈怠。其次，软件工程师需要持续的训练，以提高自己的专业能力。再者，软件工程师必须及时反馈自己的学习过程，不断改善自身的知识结构。除了刚刚学习到的基础知识外，软件工程师还需要了解新的技术、框架和工具，而且还需要不断跟踪技术的最新发展，持续地进行更新和迭代。

软件工程师的职业路径方面，可以分为实习生、初级工程师、中级工程师、高级工程师、资深工程师五个层次。实习生是最简单的路径，通常为3-6个月的时间。初级工程师一般需要7-12个月的时间，中级工程师需要15-20个月的时间，高级工程师则需要30-36个月的时间，资深工程师需要60-120+个月的时间。

# 3.基本概念术语说明
首先，要熟悉一些基本概念和术语。下面介绍几个常用的概念和术语。
## 3.1 软件工程师
软件工程师（英语：Software Engineer）是一个专门从事计算机编程、软件开发、系统集成和维护工作的专业人员。通过技术实现商业目标、解决实际问题的方式，把计算机技术应用到实际生产环境中，并且提供维护服务。其职责主要包括以下几个方面：
- 软件设计与开发：在软件工程师手里，将需求、计划和设计转换成可执行的代码。包括前端、后端、移动端、桌面端等各种开发领域；
- 软件测试：测试是软件工程师的一个重要角色，他们使用自动化测试工具和手动测试工具来验证软件是否满足用户要求；
- 软件维护：软件工程师主要关注软件的健康状态，保持软件的运行、安全、稳定和可靠性。软件维护往往需要维护工具、平台、框架和环境；
- 项目管理：软件工程师需要有良好的项目管理能力，能够将开发计划、方案、进度、资源分配以及进展落实到软件工程的每一个环节上；
- 协同工作：软件工程师的职责之一就是能够协调各种工程师、开发者和项目成员之间的关系，确保项目按时、准时、精益地完成任务。

## 3.2 概念
- 抽象模型（Abstract Model）：是对现实世界的抽象，是对计算机实现某种功能的描述，也可以指对一类问题的理解。是指现实世界的某个现象或活动的抽象概括。例如：“鸭子模型”、“银行模型”、“作家模型”。
- 敏捷开发（Agile Development）：敏捷开发是一种迭代开发方法，它鼓励快速响应客户需求，通过迭代的反馈适应市场变化和用户反馈，以短期的增量方式完成复杂的功能。
- 异步通信（Asynchronous Communication）：在没有确定的发送接收双方之前，通信就可以称为异步通信。异步通信是一种非阻塞通信方式。在异步通信过程中，发送方只管发送消息，而不会等待回复。接收方收到消息之后，可以选择处理或者保存起来，处理完毕后再回复发送方。
- 边界（Boundary）：软件设计中，通常将一个模块的行为与其他模块隔离开来，实现该模块的封装。边界即指该模块与外部环境的边界线。
- 可变性（Changeability）：软件开发中，可变性指的是当需求发生变化时，需要对软件进行修改、调试和测试的能力。可变性不仅体现在代码的可维护性上，也体现在可用性上。可用性差的软件在老旧技术的影响下，难以更新换代。
- 测试驱动开发（Test Driven Development TDD）：测试驱动开发（TDD），是敏捷开发中的一项软件开发过程，它强调对软件进行测试驱动开发。它要求所有的开发者编写一系列单元测试，然后再写业务逻辑代码。通过编写单元测试，可以确认代码的正确性。在开发过程中，程序员不断添加单元测试，确保所有代码都得到正确测试。TDD最大的优点就是可以在极短时间内获得反馈，有利于及早发现问题，从而保证代码质量。
- 代码复审（Code Review）：代码复审是指对一段代码进行检查、分析和评审，确认其符合编码标准和规范，无任何语法错误、逻辑错误、安全漏洞、性能问题等。
- 持续集成（Continuous Integration）：持续集成(CI)是一种开发实践，用来描述频繁集成代码到主干，并快速检测出失败的测试用例的开发方式。持续集成通过自动化脚本和流程，并结合一些集成工具来实现自动化构建、测试、打包、发布等一系列过程。
- 设计模式（Design Pattern）：是一些经过分类编排的、可重复使用的解决方案。它们能够在不同的情景中提供有效的参考依据。设计模式属于软件设计范畴，并不是某个具体的技术或工具。
- 文档编写（Documentation Writing）：文档编写是指编写各种文档的过程，包括需求文档、设计文档、测试文档等。它既包括技术文档、管理文档，也包括产品文档。文档是信息的载体，也是项目的权威来源。
- 集成（Integration）：集成是指将不同的软件、硬件组件、服务或系统组合成一个完整的功能。系统集成最终目的是为了提高系统的整体性能、利用率和可靠性，所以集成过程是至关重要的。
- 概念设计（Concept Design）：概念设计是指根据用户需求，建立一个或多个类或数据模型，来描述系统、产品或服务的功能特性、数据结构、交互协议等，并加以验证和定义。概念设计是过程设计的一部分。
- 微服务（Microservices）：微服务是SOA（Service Oriented Architecture，面向服务的体系结构）的一个特色，它是SOA的一种变体，是一种基于组件的架构，每一个组件独立部署、独立运行，互相之间通过轻量级的API接口互相调用。
- 模块化（Modularity）：软件设计中，模块化是指将一个复杂的系统划分为若干个小的模块，每个模块只实现一个明确的功能，彼此之间互相独立，互相配合完成系统的功能。模块化可以提高软件的可扩展性和维护性。
- 分布式（Distributed Systems）：分布式系统是指分布式系统由不同的处理单元组成，这些处理单元分布在不同的地方，彼此之间通过网络连接。分布式系统提供容错性和可扩展性，可以支持海量数据处理和超大规模并发处理。
- 负载均衡（Load Balancing）：负载均衡是提高网站或应用程序的可用性和性能的方法。负载均衡器对流量进行分发，确保服务器的负载均衡，避免单个服务器承担过多的请求。
- 前后端分离（Frontend and Backend separation）：前后端分离是指将Web应用的客户端（前端）与服务器端分离。前端负责界面呈现，用户交互，后台负责数据处理和数据访问。这样做的好处是减轻服务器压力，提高用户体验，增加了可伸缩性。
- 持续部署（Continous Deployment）：持续部署是一种软件开发实践，它频繁地将软件的最新版本部署到生产环境中，以便发现 bugs 和安全漏洞，并快速修复。
- 正交（Orthogonality）：正交是多重程度的一种级别，是指两个或多个变量在坐标系中平行或垂直。
- 服务间通信（Service to Service Communications）：服务间通信（Service-to-Service Communication）是指两个或多个服务之间相互通信的过程。服务间通信可以使用同步或异步的方式进行，同步通信要求通信双方同时返回结果，异步通信则不需要返回结果。
- 用户故事（User Story）：用户故事（User Story）是一种 Agile 方法论中，用于描述软件开发任务的形式化语言。用户故事是从用户的角度出发，描述用户为什么需要某个功能，并对这个功能产生了什么需求。用户故事以一个用户角色、一系列行为和想要实现的结果作为基准，来阐述需求。用户故事可以帮助团队成员明确需求，协同工作，达成共识，形成任务优先级和开发计划。
## 3.3 术语
- API（Application Programming Interface）：应用程序编程接口（Application Programming Interface，API）是一个允许两个以上软件应用程序进行通信的接口，它允许各个应用程序之间进行信息共享和交换。API 是计算机软件系统互相提供服务的接口，是一套协议，里面定义了调用某一功能或数据的标准，供别的程序调用。
- CD（Continuous Delivery/Deployment）：持续交付/部署（Continuous Delivery/Deployment，CD/CD），是一种 DevOps 理念下的软件开发方法论，其核心理念是：频繁、可重复地将代码提交到源代码仓库，并通过自动化测试和构建，将软件部署到测试环境、预生产环境和生产环境。
- CI（Continuous Integration）：持续集成（Continuous Integration，CI），是一种开发实践，用来描述频繁集成代码到主干，并快速检测出失败的测试用例的开发方式。
- DRY（Don’t Repeat Yourself）：不要重复 yourself （DRY），是开发人员在编码时应该遵循的原则之一。它的含义是：“不要做重复的事”，也就是说，应该避免编写重复的代码。如果代码出现需要修改的地方，应该考虑将相同的功能抽取出来，这样就能避免重复的代码。
- KISS（Keep It Simple Stupid）：保持简单，保持傻瓜 (KISS)，是指尽可能的使自己的设计保持简单、易于理解。具体体现在，不要过度设计、过度抽象、避免冗余。
- Linter（Linting tool）：Linting 是指识别和纠正代码中的错误、 bugs 的程序。它与代码审查一样，是防止代码质量低下的有效方式。
- OOP（Object-Oriented Programming）：面向对象编程（Object-Oriented Programming，OOP）是一门编程范式，它以类、对象、继承、封装、多态等概念为基础，通过类和对象的方式来组织代码，并通过数据抽象、信息隐藏来实现程序的重用。
- RFC（Request For Comments）：征求意见书（Request for Comments，RFC）是一个由互联网社区开发者提交的技术文件，它记录了对互联网相关技术领域中重要问题的建议和想法。RFC 是互联网业内的一个重要社区贡献。
- SOLID（Single Responsibility Principle / Open–Closed Principle / Liskov Substitution Principle / Interface Segregation Principle / Dependency Inversion Principle）：单一职责原则、开闭原则、里氏替换原则、接口隔离原则、依赖倒置原则。SOLID 是 Object-Oriented Programming（OOP） 中用于创建松耦合、可维护和可扩展的设计原则集合。
- TDD（Test-Driven Development）：测试驱动开发（Test-Driven Development，TDD）是敏捷开发中的一种软件开发方法，它强调通过编写测试案例来推动开发过程，目的是让开发人员编写代码之前先编写测试代码。
- UML（Unified Modelling Language）：统一建模语言（Unified Modeling Language，UML）是一门基于标准图表符号的语言，它提供了一套标准的表示方法，用来图示面向对象、静态、动态和数据库等方面的模型。UML 中的元素包括类、对象、关联、泛化、实现、用例等。
