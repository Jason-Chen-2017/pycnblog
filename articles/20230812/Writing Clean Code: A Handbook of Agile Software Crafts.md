# 1.简介
  

“编写整洁的代码”一书的作者Eric Evans曾说过一句名言——“代码质量决定于开发人员是否能够持续不断地改进它。”很多程序员都认为阅读并理解优秀代码至关重要，但更多时候他们只是将其视作一种习惯或手段。要想提高代码质量，除了阅读优秀代码外，还需要付出更大的努力，不断学习、反复实践、总结经验和模式。而本书正是作者对这一主题的探索之作。

《1. Writing Clean Code: A Handbook of Agile Software Craftsmanship》全书共分9章，分别从命名规范、注释风格、编程规范、SOLID原则、设计模式、异常处理、单元测试、集成测试、部署发布等方面介绍了程序员应该具备的编码能力。其中每一章的内容都有详细的代码实例和解决方案供读者参考。

由于这本书是为软件工程师写的，所以作者根据自己的实际工作经历、团队成员及工具配置等，提供多种示例，帮助读者理清编程思路，快速上手。另外，本书不止适合软件工程师阅读，也适合所有软件行业的技术人员阅读，因为它涉及到代码质量方面的知识。本书适合作为培养新一代软件工程师的指导手册，希望能够帮助读者深入理解软件设计、开发、测试和部署中的最佳实践和方法论。

# 2.基本概念、术语及缩写释义
- 文件(file)：计算机中存储信息的最小单位，如程序源文件、数据库文件、图像文件等。
- 函数(function)：一个模块化的、可重复使用的、有输入输出的程序片段。
- 类(class)：用来描述具有相同属性和行为的数据结构。
- 方法(method)：定义在类的内部的函数，用于实现数据和功能之间的交互。
- 命名规则：
  - 使用易于理解的单词：名称应尽可能明确且易于理解。
  - 用驼峰命名法：如MyFirstClass。
  - 用动词或名词开头：如showDialog()。
  - 在同一作用域内保持唯一性：如getName()。
  - 只使用ASCII字符：避免使用中文、日文、韩文字符。
  - 不用拼音：除非是专业术语。
  - 使用描述性的名称：如getCustomerByName()比getByName()更容易理解。
- 注释风格：良好的注释能够让别人更轻松地了解你的代码，包括你自己和其他程序员。良好的注释应该提供足够的信息，让别人能够准确地理解你所写的东西。注释不仅可以让代码更加易懂，而且还会减少错误、漏洞和缺陷。
  - 块注释（Block Comments）：用大括号{}包围的一系列文本，通常在代码前后位置。建议使用多行形式的块注释。
  - 行注释（Inline Comments）：以//开头的一行注释，通常只出现在独立一行代码前。
  - 提供文档：如果没有注释，那就不要编辑代码了！
- SOLID原则：设计模式的五项原则之一。主要是用来降低代码的耦合性和可扩展性。它包括以下五个原则：
  1. Single Responsibility Principle (SRP): 单一责任原则（SRP）又称单一功能原则（SFPO）。该原则规定一个对象（类、模块、函数等）应该有且只有一个引起它的变化原因。换句话说，一个类只负责完成当前领域的业务逻辑。
  2. Open/Closed Principle (OCP): 开闭原则（OCP）表示一个软件实体（如类、模块、函数）应该对于扩展是开放的，但是对于修改是关闭的。换句话说，在不修改原有代码的情况下，可以通过增加新的代码来扩展功能。
  3. Liskov Substitution Principle (LSP): 里氏替换原则（LSP）由加利福尼亚大学计算机科学系的John Liskov提出。它表明子类型必须能够替换它们的基类型。换句话说，继承关系被设计为尽可能广泛，使得父类型的代码也可以正常运行。
  4. Interface Segregation Principle (ISP): 接口隔离原则（ISP）即客户端不应该依赖它不需要的方法。换句话说，就是设计接口时要小心谨慎，只暴露给客户端必需的接口。
  5. Dependency Inversion Principle (DIP): 依赖倒置原则（DIP），主要内容是高层模块不应该依赖底层模块，二者都应该依赖其抽象；抽象不应该依赖细节，细节应该依赖抽象。换句话说，在程序设计中，变量的声明类型（如接口或抽象类）应该高于变量的实际类型。
- 设计模式分类：根据创建对象的目的、作用及变化方式，设计模式分为三大类：创建型模式、结构型模式和行为型模式。
  - 创建型模式：主要关注如何创建对象，包括单例模式、建造者模式、工厂模式、原型模式。
    - 单例模式：保证一个类只有一个实例，也就是一个进程只能有一个实例。
    - 建造者模式：把复杂对象的创建过程封装起来，通过一步一步构造最终得到一个完整对象。
    - 工厂模式：提供一个创建对象的接口，隐藏创建对象所需的逻辑，用于替代new运算符。
    - 原型模式：用原型实例指定创建对象的种类，并且通过拷贝这些原型创建新的对象。
  - 结构型模式：主要关注对象之间的组合关系，包括适配器模式、桥接模式、装饰器模式、组合模式、享元模式。
    - 适配器模式：将一个接口转换成客户希望的另一个接口。
    - 桥接模式：将抽象与实现分离，使他们可以独立变化。
    - 装饰器模式：动态地添加一些职责到对象中。
    - 组合模式：将对象组合成树形结构以表示“部分-整体”的层次结构。
    - 享元模式：运用共享技术有效地支持大量细粒度的对象。
  - 行为型模式：主要关注对象之间进行通信的方式，包括观察者模式、模板方法模式、命令模式、策略模式、状态模式、职责链模式。
    - 观察者模式：多个对象间存在一对多依赖关系，当一个对象改变状态时，将自动通知其他对象，从而达到通讯的目的。
    - 模板方法模式：定义一个操作中算法的骨架，而将一些步骤延迟到子类中。
    - 命令模式：将一个请求封装为一个对象，从而使你可以参数化其它 objects ，延迟或取消请求，支持撤销操作。
    - 策略模式：定义了一系列算法，将每个算法封装起来，并让它们可以相互替换。
    - 状态模式：允许对象在内部状态发生改变时改变它的行为，对象看起来似乎修改了它的类。
    - 职责链模式：使多个对象都有机会处理请求，从而避免请求的发送者和接收者之间的耦合关系。
