# 1.简介
  

在软件工程领域，代码质量是一个非常重要的问题。为了提升软件开发的效率、质量和进度，软件工程师们经历了漫长而曲折的学习过程。其中Clean Code这个概念最早由“Clean Code”一书提出，被许多著名的软件工程大牛倡导。
现代编程语言已经提供了很多便利的语法特性，使得代码的编写变得更加简单、高效、可读性更好。但仍然需要保证代码的健壮性，不引入运行时错误，正确处理各种异常情况等，这就需要工程师们重视代码的可维护性。过去几年里，越来越多的软件工程师参与到敏捷开发中，习惯于重构代码、自动化测试、持续集成等技术，并逐渐形成了一套完整的软件开发流程。因此，Clean Code也成为敏捷开发中的一个重要主题。

《Clean Code: A Handbook of Agile Software Craftsmanship》（中文译本《干净的代码：企业级软件匠艺》）正是这样一本书，它从代码规范，命名规则，函数设计，注释等多个方面进行了详细阐述，试图提供给软件工程师和管理者一些有益的经验。阅读本书可以帮助大家理解软件工程中常见的软件危害、代码坏味道、编码实践，以及如何有效地构建软件产品。

# 3. 目录
## 一、代码风格
### 1.1 代码缩进与空格
代码缩进应该采用统一的四个空格，并且一定不能混用TAB键与空格。

### 1.2 文件名称
文件名应当全部小写，单词之间以中划线分隔。

### 1.3 函数/变量命名
函数和变量名都要见名知意，能够准确反映它的含义。

### 1.4 类/接口命名
类的命名要体现其职责或功能，如Manager, Processor, Helper, Controller之类。接口命名要充分考虑其功能性。

### 1.5 方法命名
方法命名要体现其职责或功能，一般来说，动词或名词+动词或者名词+名词+动词都是可以的。

### 1.6 常量命名
常量全大写，单词间以下划线连接。

### 1.7 行宽
每行代码字符数不要超过80个，超过的部分必须使用换行符分割。

### 1.8 括号与空格
所有的左右括号、花括号、中括号前后都要加上空格。

### 1.9 可读性与易懂性
代码写作要保持足够的可读性，代码尽量使用简单、明了、直观的方式表达出来。

### 1.10 使用描述性变量名
变量名要尽可能地具备描述性，能够清楚地表明变量所表示的内容。

## 二、注释
### 2.1 注释的意义
好的代码注释能够帮助我们记住代码的作用，让别人更容易理解我们的代码。好的注释往往有助于团队沟通，减少出错，提高代码的可读性。

### 2.2 文件头注释
每个源文件开头都应该包含一个文件头注释，包括作者姓名、创建时间、版本信息等。

### 2.3 函数注释
函数的注释应当对函数的输入参数，返回值，作用，边界条件，以及非正常情况进行说明。

### 2.4 类注释
类注释应该对类的功能，属性，方法，依赖关系及其作用范围进行说明。

### 2.5 数据库字段注释
数据库字段的注释应该能够准确反映数据库中字段的数据结构。

## 三、编程原则
### 3.1 SOLID原则
SOLID原则是5个主要原则之一。它的目的是通过对软件设计的约束和封装来建立稳定的、可预测的软件。

S: Single Responsibility Principle (单一职责原则)
只有一个引起变化的原因，也就是只有一个变化的地方，同时只需做该做的事情即可。

O: Open/Closed Principle （开闭原则）
软件实体(类、模块、函数等)应该对扩展是开放的，对修改是关闭的。

L: Liskov Substitution Principle (里氏替换原则)
任何基类可以在其子类中替代它的功能，因此，所有引用基类的地方无需修改也照样可以工作。

I: Interface Segregation Principle (接口隔离原则)
使用多个专门的接口比使用单个庞大的接口要好，即客户不需要知道太多不需要的方法。

D: Dependency Inversion Principle (依赖倒置原则)
高层模块不应该依赖低层模块，二者都应该依赖抽象；抽象不应该依赖细节，细节应该依赖抽象。

### 3.2 DRY原则
DRY原则即Don't Repeat Yourself。该原则认为同一件事应该只存在一个地方，避免重复造轮子。

### 3.3 KISS原则
KISS原则即Keep It Simple and Stupid，中文翻译为“保持简单”，就是说你只需要做最简单的事情，不要做太多的事情。

### 3.4 YAGNI原则
YAGNI原则即You Ain’t Gonna Need It，中文翻译为“用不着”。这个原则是说在设计一个软件时，你应该先确认哪些功能是必要的，而不是那些功能是不必要的。
