# 1.简介
  

随着云计算和云服务的广泛应用，越来越多的人开始关注OpenStack技术平台的发展。OpenStack是一个开源的软件项目，它提供了一个基于虚拟化技术、网络结构、自动调度和管理的软件定义的基础设施即服务（IaaS）平台，帮助企业构建私有云和公共云。截止目前，OpenStack已成为全球最重要的云计算平台之一。目前，有大量公司已经将OpenStack作为内部部署的基础架构或服务来运行，有的甚至为了盈利甚至商业化，不惜以公开源代码的形式分发其内部软件。那么，什么是完全开放源代码？为什么OpenStack的代码一直没有完全开放？又该如何才能做到这一点呢？本文从相关的研究论文出发，详细阐述了OpenStack源码的开放状况及其原因，以及在云计算领域应该采取的行动，希望能够对读者有所启发。
# 2.OpenStack基本概念
OpenStack是一个开源的软件项目，它提供了一套完整的软件组件集合，包括计算、存储、网络等功能模块，用来构建私有云和公共云。OpenStack项目由OpenStack基金会（OSF）社区发起，采用Apache License版本2.0发布。OpenStack中主要组件包括Nova、Neutron、Swift、Keystone、Glance、Cinder、Horizon、Ceilometer、Heat等。其中，主要服务组件包括Nova（提供计算服务），Neutron（提供网络服务），Swift（提供对象存储服务），Keystone（提供身份认证、授权和管理服务），Glance（提供镜像仓库服务），Cinder（提供块设备服务），Horizon（提供UI界面），Ceilometer（提供监控服务），Heat（提供资源编排服务）。如下图所示。
# 3.OpenStack源代码开放状况及其原因
虽然OpenStack的官方网站上明确声明其代码是Apache许可，但OpenStack项目官方仍然没有完全开放源代码。在国内外的一些报道中，有很多读者抱怨OpenStack源码开放的问题。例如，张宏旭（ID：thepowerofme）团队指出，由于公司内部的各种限制，比如政策法规禁止分享某些源码，导致OpenStack项目的社区不能完全自由地接触OpenStack源码。据悉，OpenStack基金会开发者计划于今年年底前完成对源代码的完全开放，而且项目的领导层也同意将会在Github上开放OpenStack的所有源码。另外，Hadoop、Kubernetes、Mesos等开源系统也提供了大量的源代码，它们也可以被开源社区免费使用。相比之下，OpenStack作为中国的顶级开源项目，自身有很强的知名度，更需要通过官方途径让更多的人参与到开发的过程中，因此才引起一些非议。
# 4.OpenStack Open Source Code Policy
1. The initial development of the OpenStack project will take place on GitHub and all subsequent releases of code will be released under Apache-2.0 license. All patches submitted to OpenStack will also need to be released under this same license. 
2. Community members are free to access the OpenStack codebase on Github at any time. Any community contribution is welcomed and will be reviewed by the core team before merging into the master branch. 
3. All contributions must follow the guidelines for submitting changes to OpenStack: bug fixes should include tests that prove the problem exists (and do not regress), new features should have both functional and unit tests as well as documentation, and any change in behavior or user interface should be accompanied with a deprecation path if possible. Patches should only contain one feature or bug fix each, unless they depend on another patch. Changes should generally go through the review process and be held up to standards set out in the governance documents. When proposing changes, please use our Bugzilla tool, following the process described here. For more information, refer to http://docs.openstack.org/infra/manual/developers.html. 
4. In order to encourage maximum collaboration amongst developers, the project will adopt several mechanisms to ensure that contributions from various organizations are reviewed fairly and consistently: a set of Committer roles that have write access to the repository, an organizational goal to increase diversity within the project, and a transparent voting system where contributors can express their opinions and influence decisions about the direction of the project. This approach aims to create an environment where developers feel comfortable making incremental improvements without having to worry about who gets to decide what. While this may initially slow down some contributions, it ensures that the project stays healthy and thriving over time. Additionally, these mechanisms will help to reduce conflicts of interest within the project by ensuring that different participants can make different types of contributions without undue influence.
