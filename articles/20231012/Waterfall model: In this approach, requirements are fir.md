# 1.背景介绍


Waterfall模型（以下简称WF）是一种项目管理方法，其基本思想是将整个开发过程分成几个阶段并严格按照固定顺序进行。首先，进行需求分析，确定系统要实现什么功能以及需要哪些材料。然后，制定产品或服务的初始设计计划，包括规划、需求、设计、编码、测试等各个环节的任务，并确保按照计划完成每一个环节。最后，根据测试结果对系统进行维护和改进，直到满足最终用户的需求。

特别指出，在WF模型中，无论是否采用面向对象技术，其核心是“文档驱动”，即所有的活动都要严格按照流程图进行记录和管理。因此，WF模型注重文档的完整性、准确性和完整性。它的优点是简单易懂，缺点是效率低下，往往无法应对复杂的需求变更或竞争对手的反馈。

# 2.核心概念与联系
本节从项目开始至结束，梳理WF模型中的一些关键概念和联系。
## 2.1 概念
- 阶段：开发过程中的一个重要阶段，由多个子阶段组成，如需求分析、设计、编码、测试等。每个阶段有明确的目标和时间表。
- 文档：在WF模型中，每一个阶段都要产生相应的文档，如需求文档、设计文档、编码文档等。这些文档必须详细地阐述相关信息，但不允许随意删改。如果出现文档错误或遗漏，会影响后续工作的正确性。
- 流程图：流程图是一个文档，描述了一个阶段的活动清单，它可以帮助团队成员了解活动的先后次序、时间和人员。流程图可用来检查流程是否合理、顺利进行，还可以发现文档中的不足之处。
- 迭代：在WF模型中，“迭代”通常是一个重要的过程。在迭代过程中，新版本的软件或服务被交付给最终用户，获得反馈，并根据反馈进行调整和修改，以提高质量。迭代过程可以有效避免单点故障，也能提升开发速度。
- 风险管理：在WF模型中，风险管理也是极其重要的一环。它通过评估系统、工具和环境的可用性、安全性、可靠性、性能及其他相关因素，确定可能发生的各种风险，并采取相应措施减小这些风险。
## 2.2 联系
1. WF模型属于静态的项目管理方法。不需要频繁的迭代和交流，适用于短期项目或特定类型的软件系统。

2. 在WF模型中，文档的作用是保证系统的可理解性和准确性。如果文档不清晰、不准确、不全面，会造成后续工作的延误和损失。因此，要求对文档的编写和管理保持高度的投入精力。

3. 如果没有文档驱动，则WF模型将会成为过时的“瀑布模型”。只有“文档驱动”才能保证项目的成功。

4. 由于WF模型强调文档驱动，因此它有着良好的实践经验积累，并且具有坚实的基础。

5. “迭代”是WF模型的一个重要机制，它鼓励及时反馈、承担责任、快速迭代。因此，迭代的次数越多，出现的问题就越少。

6. WF模型注重风险管理，它评估了项目可能存在的各种风险，并进行预防措施，确保项目顺利完成。
