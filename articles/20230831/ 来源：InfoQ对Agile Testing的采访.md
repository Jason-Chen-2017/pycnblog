# 1.简介
  


Agile testing (另译为敏捷测试)是一个新兴的测试方法论，它提倡通过迭代和交互的方式进行测试，从而更好的满足业务需求、客户的要求和产品的进化。

Agile testing把测试作为一种软技能，强调测试人员应该具备的敏捷意识。相对于传统测试方法中所有的测试工作都集中在一个测试团队里面，Agile testing鼓励小型的测试团队可以自主地解决各种各样的问题。因此，Agile testing能够更有效地响应快速变化的市场需求。

# 1.测试模型及其演变历程
## 测试模型演变历程
### 测试驱动开发（TDD）
> 在 2003 年， TDD 模式被提出，由 <NAME> 提出，他认为单元测试应该在开发过程中就完成，而不仅仅是在测试之后才进行。在 2007 年， John Glass 等人对 TDD 的理念进行了证实，并提出了一套自动化测试框架。现在，每当一个开发者编写一个新的功能模块或修复了一个 bug 时，都会先编写相应的测试用例。当然，编写测试用例是一个时间上耗费精力的过程，但这是值得的。

### 测试金字塔模式
> 2007 年， Martin Fowler 等人提出了“测试金字塔”模式。测试金字塔由四层构成：UI 层、逻辑层、数据层和系统层。越往上，测试覆盖的范围越广。从底部向上的顺序为 UI 测试、端到端测试、单元测试、集成测试、系统测试和接口测试。但是，由于这样的测试方案过于复杂，所以一般不会使用测试金字塔，取而代之的是结构化编程模式。

### 演进式设计（XP）
> 2001 年， <NAME> 和 <NAME> 联合创立了 “Extreme Programming Excellence（极限编程卓越）” 方法论，并把测试作为其中不可或缺的一环。2005 年，该方法论成为国际标准，其特色就是采用 XP 宣言以及其相关实践。 XP 强调快速反馈、价值分配、简单性、协作性等，并通过重视“测试自动化”和“反馈循环”等核心概念，使得软件开发更加可靠、高效、稳定。

### 测试驱动面向对象开发（TDO）
> 在 2009 年，<NAME> 等人提出了 TDO （Test Driven Object-Oriented Development）的方法论。TDO 鼓励开发者在编码之前编写测试用例，然后根据这些用例驱动实现对象。这种方法在保证正确性、可维护性和可扩展性方面都非常有益处。2010 年，TDO 方法论成为国际标准，并被广泛应用。

### 测试驱动结构化编程（TDSP）
> 2011 年，Dave Brubeck 提出了 TDSP 方法论。TDSP 是基于 TDD 的结构化编程方法论。它首先考虑到如何建模、组织和管理代码，然后再编写测试用例、实现代码和运行测试。TDSP 通过引入结构化编程的一些关键概念如结构化设计、接口隔离和测试驱动开发，提升了代码质量和可维护性。

### 测试驱动开发（TDD）与 TDSP 的区别
> 两者都侧重于测试驱动开发和测试驱动结构化编程中的核心概念。但是，TDD 更关注代码质量而不是整个流程的可持续性，即便在迭代的过程中也会频繁地回归测试。相比之下，TDSP 更关注流程的可持续性，它的每一步都是为了达到目标而设计的，不断迭代和改进，最终实现“测试即文档”，让每个开发者都能清楚地理解代码背后的意图和目的。

## 为什么需要敏捷测试？

### 软件开发的复杂性
> “如果你的软件是一个简单的四则运算器，那么你可能不需要做很多测试；但如果你正在构建一个具有复杂功能的数据库应用程序，你需要花费大量的时间、精力和资源来编写、维护和维护测试代码。”——<NAME> 

现实世界的软件系统比起复杂的四则运算器来说更加复杂，而且还经常会随着时间的推移、功能的增加和变化而演进，因此，软件测试也会变得更加困难。

### 测试的复杂性

敏捷测试倡导测试工作以周期短、反馈及时为特征，并提出了“测试即文档”的观点。但是，编写测试用例本身仍然是一项艰巨的任务，因为测试人员不仅要懂得需求、业务规则、边界条件、输入输出等，还要掌握各种各样的测试技术，包括单元测试、集成测试、端到端测试、压力测试、稳定性测试等。因此，测试人员需要不断学习新的技术和工具，才能编写出全面的测试用例。

### 跨部门协作与动态环境下的复杂性

敏捷开发方法论的最大优势在于它鼓励多方参与开发，而且支持持续集成和部署（CI/CD）。因此，测试工作也将受到不同部门的关注，包括设计、开发、测试和运维等多个角色。

在敏捷开发的过程中，需求会不断变化，这使得测试工作变得更加困难。在敏捷开发的环境下，需要频繁地交付软件更新，包括手动、自动和半自动的测试。同时，测试工作还需要与其他团队的沟通，包括制定测试计划、跟踪测试结果、制订测试策略和工具等。因此，测试工作的复杂性也逐渐增加。

# 2.敏捷测试的优点
1. 节约时间和资源

   在测试过程中，可以通过测试金字塔模式进行快速定位，并且可以根据不同阶段的错误类型进行分类，降低测试成本。此外，敏捷测试方法论鼓励反复测试，在软件的开发生命周期内保持最大程度的适应性，以发现更多的bug。

2. 提升代码质量

   敏捷测试倡导在编码前编写测试用例，从而在软件测试过程中更容易发现潜在的问题。测试人员还可以使用各种测试技术，比如单元测试、集成测试、端到端测试、压力测试、稳定性测试等，来验证代码的质量。此外，敏捷开发方法论鼓励按计划编写测试用例，并提倡重构、持续改进，以提升代码质量。

3. 促进沟通和协作

   在敏捷测试方法论中，测试工作可以更加独立、自治，而且可以以项目的形式进行协作。测试人员可以在不影响其他团队工作的情况下进行测试，以便达到增强信心、减少风险、促进工作流的目的。

4. 保证软件可靠性

   敏捷测试的重要特点之一是测试持续集成和部署（CI/CD），它可以帮助开发人员检测和隔离错误，并在项目结束时快速找到并修复它们。此外，它还可以帮助减轻测试人员的压力，并给予他们更多的时间来创造更好的测试用例。

5. 提升软件质量

   敏捷开发方法论鼓励软件工程师采用精益开发方式，包括优先级排序、重构和持续改进，从而提升软件质量。同时，它也鼓励测试人员为改善测试质量提供意见，从而提升整体性能。

# 3.敏捷测试中最重要的三个原则和六个要素
## 三原则
敏捷测试方法论共计七条原则，包括：

### 1. 可重复性：

所有测试都应该是可重复执行的，这样就可以保证测试的有效性、准确性、及时的响应能力。测试人员应该编写测试脚本、配置文件或配置参数，使测试环境和测试用例可以在任意机器或环境中运行，且结果一致。

### 2. 有价值度量：

测试工作应该遵循“有价值度量”，只有测试所产生的价值才会被认真对待。测试工作的目标不是去找那些看起来很难解决的错误，而是找到那些真正值得去修复的错误。

### 3. 小步快跑：

测试工作应该以小步快跑的方式进行，也就是尽可能地缩短开发周期。测试人员应该对小的代码改动或较小的测试用例，经常进行回归测试，确保测试通过后才提交至版本库。

## 六要素
敏捷测试又分为六个要素，分别是：

1. 测试模型

   敏捷测试通常采用TDD、BDD、ATDD（验收测试驱动开发）、RTDD（回归测试驱动开发）、AutoDD（自动化测试驱动开发）等测试模型。在实际应用中，需要结合具体情况选取合适的测试模型。

2. 测试策略

   敏捷测试的策略包括“测试即文档”、“测试边界”、“测试抽象”、“测试自动化”、“测试分类”等。测试人员应该多花时间思考测试策略，通过编写测试用例、测试计划、测试脚本来实施测试策略。
   
   此外，敏捷测试方法论也提倡采用“需求确认测试”、“回归测试”、“负载测试”、“兼容性测试”、“冒烟测试”等测试策略，以确保软件的质量。

3. 测试环境

   测试环境应该是自动化的，以避免因测试环境的差异导致测试失败。测试环境包括测试机器、测试数据库、测试网络等，应该使用持续集成工具进行自动部署和更新。

4. 测试人员

   测试人员应该全面掌握测试技术，包括单元测试、集成测试、端到端测试、压力测试、稳定性测试等。测试人员应该熟悉各种测试工具，包括单元测试框架、测试报告生成工具等。

5. 测试工具

   测试工具可以帮助测试人员快速编写测试用例、生成测试报告、监控测试进度、分析测试结果。测试工具包括各种类型的自动化测试工具、手动测试工具等。

6. 测试数据

   测试数据是通过测试用例和测试场景构造出来的，测试数据的准确性决定了测试的准确性。因此，测试数据也是敏捷测试不可或缺的一环。
