# 1.简介
  

作为数据处理中心、信息安全部门的基础设施建设者，在研发、运营和维护大量数据的同时，不得不面对复杂、多样化的各种数据源。如何存储这些数据，确保其安全性与可用性？如何快速准确地查询、分析和挖掘数据？以及其他相关数据管理技术，都属于数据库领域重要的研究和开发工作。

数据库系统发展历史可以分为两个阶段——第一阶段是单机数据库系统，第二阶段是分布式数据库系统。目前市场上主流的数据库产品主要集中在第二阶段，包括Oracle、MySQL等关系型数据库产品、HBase、MongoDB等NoSQL数据库产品和Cassandra、Redis等键值对数据库产品。其中，关系型数据库产品Oracle、MySQL的优点在于结构化的表结构，灵活的数据类型支持以及完善的SQL接口使得其易用；而NoSQL数据库产品HBase、MongoDB更具有高扩展性、高性能、可容错性的特点。

本文将介绍关系型数据库产品Oracle、MySQL、NoSQL数据库产品HBase、MongoDB的原理、特征、适用场景以及典型应用场景。

# 2.关系型数据库的概念和特性
## 2.1 什么是关系型数据库（RDBMS）？
关系型数据库管理系统（Relational Database Management System， RDBMS），是指采用了关系模型组织数据的数据库管理系统，其特点是功能完整性、数据一致性和并发控制容易保证数据正确性。其由一个或多个关系组成，每个关系是一个表格，记录着相互之间存在的一对多、多对多或一对一的关联关系。关系型数据库管理系统管理着关系型数据库，用来存储、检索和更新企业内在的各种复杂的信息。

关系型数据库管理系统分为层次式数据库系统和基于网格的数据库系统两种。前者是存放在磁盘上的，后者则存放在网络服务器上。按照结构复杂程度的不同，关系型数据库分为事务型数据库、基于方块的数据库、对象/实体关系数据库、层次型数据库、集合/集合包含的数据库和空间数据库。

关系型数据库最初是由 IBM 的丹·卡迈克尔·梅耶 (<NAME>) 和丹·唐纳德·范罗苏姆 (<NAME>) 在上世纪70年代设计开发，最早的名字叫做 "IBM 7090"，但随着时间的推移逐渐被称作关系型数据库管理系统。20世纪90年代初期，随着关系型数据库的普及，它成为构建任何规模企业应用的必备工具。如今，关系型数据库已经成为事实上的标准，并且仍然是许多大型公司使用的首选数据库。

## 2.2 关系型数据库的特性
### 2.2.1 数据模型
关系型数据库中的数据以表的形式存在，表中每一行代表一条记录，每一列代表字段。表与表之间存在外键关系，即一个表的某个字段值等于另一个表的主键值，这样就可以建立起联系。关系型数据库提供了丰富的数据类型，包括整型、浮点型、日期时间、字符型、布尔型等。

关系型数据库支持完整性约束，包括实体完整性、参照完整性、用户自定义完整性、单属性唯一性、多属性唯一性、以及业务规则约束。关系型数据库通过定义视图，能够看到相关联的多个表的全部数据。

### 2.2.2 查询语言
关系型数据库中支持的查询语言有SQL语言和过程化编程语言，如PL/SQL、T-SQL等。SQL语言用于定义、插入、更新、删除数据库中的数据，并进行条件查询、排序、分组、聚合等操作；过程化编程语言用于定义函数、触发器等。

### 2.2.3 并发控制
关系型数据库通过锁机制实现并发控制。对于读操作来说，如果不同的查询和事务需要访问相同的数据，那么数据库会对其加锁，防止其他事务的修改和提交。对于写操作来说，数据库允许多个事务同时访问相同的数据，但是只有持有锁的事务才可以进行修改，其他事务必须等待锁释放才能进行修改。当多个事务修改相同的数据时，数据库会自动生成冲突检测，以解决并发冲突。

### 2.2.4 ACID特性
关系型数据库具备ACID特性，该特性描述了一组用于保持数据一致性的规则，包括原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）、持久性（Durability）。

 - 原子性（Atomicity）：事务是不可分割的，要么全部执行成功，要么全部失败回滚。
 - 一致性（Consistency）：事务只能在事务开始之前或者结束之后才有效，这表示一个事务从一个状态到另一个状态必须遵循严格的规则。
 - 隔离性（Isolation）：一个事务的执行不能被其他事务干扰。
 - 持久性（Durability）：一旦事务提交，其结果将永久保存到数据库中。

### 2.2.5 性能
关系型数据库在处理海量数据时表现出色，具有很好的读写效率。由于关系型数据库的数据存放在磁盘上，因此读取速度快，尤其适合高频的查询操作。由于采用B+树索引，数据库读取速度较快，这也是关系型数据库选择B+树索引的原因之一。

## 2.3 NoSQL数据库
NoSQL（Not Only SQL）是一个新的概念，指非关系型数据库。传统关系型数据库以行和列的数据结构存储数据，NoSQL数据库却以文档、图形或对象的方式存储数据。NoSQL数据库可以说是一种类JSON的键-值对存储数据库。

NoSQL数据库与传统的关系型数据库相比，主要区别在于它的关注点不是如何将数据结构转换成数据库表，而是如何在一个数据集中存储、查询和索引数据。NoSQL数据库的目标是使得数据更容易存储、访问和处理，而不是像关系型数据库那样将数据结构映射到数据库表。NoSQL数据库中最流行的三种NoSQL数据库分别是HBase、Couchbase和MongoDB。

### 2.3.1 HBase
Apache HBase (TM)是一个开源的分布式NoSQL数据库，它能够提供高伸缩性和高可用性。HBase由Google的研究人员开发出来，是谷歌大规模分布式搜索和分析系统的基础。它完全面向列的结构化数据存储，并提供随机、实时的查找能力，适用于那些要求高吞吐量、大容量和低延迟的数据访问。

HBase的基本单位是表（Table），表由列族（Column Family）组成，每列族又由若干列组成。表中的每条记录都是由多个列簇的值组成，这些值存储在一起，但实际上可以从不同的列簇中检索数据。HBase的数据模型非常类似于Bigtable。

HBase提供高性能的读写操作，一次查询请求可以涉及多个表的数据，并将其组合起来返回给客户端。HBase支持海量数据存储，由于HBase使用Thrift作为客户端接口协议，无需费心去解析复杂的数据结构，所以HBase的性能比较优秀。HBase还有一个特性就是容错性高。当一个节点出现故障时，只要集群还有剩余的节点，HBase就可以继续运行。

### 2.3.2 Cassandra
Apache Cassandra 是一款开源分布式 NoSQL 数据库，它最初是由 Facebook 的斯坦福大学的研究人员开发的。Cassandra 使用类似 Bigtable 的数据模型，但它还提供了一些独有的特性。例如，它支持按键范围查询、动态负载平衡、轻松扩展以及对复制方案的自我检测。

Cassandra 支持的核心数据结构有：

- 键值存储：Cassandra 将数据存储在 keyspaces 中，其中每个 keyspace 包含多个 tables。每张 table 可以理解为一张数据库表，key 是索引、行标识符，value 是数据。

- 列族：Cassandra 将同一个 table 中的数据划分为多个列族，每个列族具有自己的数据类型。例如，可以使用不同的列族来存储不同的时间戳或版本号。

- 自动平衡：Cassandra 可以自动重新平衡数据分布，以均匀分配数据到各个节点上。这意味着即使在增加或减少节点数量时，集群仍然保持良好的性能。

- 分布式查询：Cassandra 提供对数据分布式查询的支持。查询可以在所有副本、子群集、节点、集群或所有数据上面运行。

Cassandra 以“高可用性”和“一致性”著称，并且提供了数据完整性的保证。Cassandra 不仅支持动态水平扩展，而且还可以配置复制策略，以便在不同的地方复制数据，提高系统的可靠性。

### 2.3.3 MongoDB
MongoDB是一个基于分布式文件存储的NoSQL数据库。它是一个面向文档的数据库管理系统，最大的特点是提供高性能。MongoDB把数据存储为一个文档，它是一个动态的面向对象的数据库。字段值可以包含其他文档、数组及二进制数据。除了支持SQL查询，还提供了自己的查询语言。

MongoDB的所有数据结构都是BSON格式的，一个文档可以嵌套其他文档。文档中的每一个字段都有自己的类型，MongoDB可以对不同的类型的数据进行索引。Mongo支持跨文档的事务，这使得在文档级别上的原子性操作成为可能。

MongoDB支持Sharding和ReplicaSet功能，这可以让集群横向扩展。它支持丰富的查询语言，包括对数组及文本的全文搜索功能。此外，MongoDB还有很多第三方插件及工具包。

# 3.Oracle数据库
## 3.1 Oracle的介绍
Oracle是一家高科技公司，目前主要提供关系型数据库服务。其商标是皇冠状物种的形象，也就是“橙果”。Oracle是世界上第一个拥有专利的数据库厂商，总部位于美国旧金山，主要生产企业级数据库软件。目前，Oracle已经成为全球最流行的数据库厂商，主要应用领域包括航空航天、银行、保险、零售、电信、制造、教育、高科技等。

## 3.2 Oracle的基本概念
### 3.2.1 用户（User）
用户指的是数据库管理员或应用程序开发者，他们利用Oracle数据库产品，创建、修改、查询数据库中的数据。在创建新数据库时，需要指定数据库的用户名和密码。

### 3.2.2 数据库（Database）
数据库是用于存放数据的仓库，它包含了一系列的表（Tables）、视图（Views）、索引（Indexes）、存储过程（Stored Procedures）、序列（Sequences）等构成。

### 3.2.3 表（Table）
表是由记录（Records）和字段（Fields）组成的数据结构。一个表通常对应于某个特定类型的数据，例如存储员工信息的表，就包含姓名、地址、电话号码等字段。

### 3.2.4 记录（Record）
记录是表中的一条数据项，它包含多个字段。每条记录都有唯一的标识符（Primary Key）。

### 3.2.5 字段（Field）
字段是表中的一个数据单元。它有字段名称、数据类型、长度、精度和默认值。

### 3.2.6 索引（Index）
索引是一个帮助数据库快速找到数据的结构。它根据一个或多个字段建立，并存储在表的特定区域。索引可大大提升数据库的查询效率，但也降低了其插入、更新和删除的效率。

### 3.2.7 视图（View）
视图是一个虚拟的表，其内容由其他表的内容生成。视图是数据库中虚构出来的一个表，其作用类似于一个电子透镜，将一张真实的表或者多张表的内容全部映射为一个虚拟的表，用户可以通过这个虚拟的表查看数据。

### 3.2.8 存储过程（Stored Procedure）
存储过程是一个预编译的数据库操作，它封装了一系列操作，并存储在数据库中。存储过程使得数据库用户可以实现复杂的逻辑和运算，并可以将此过程重用。

### 3.2.9 函数（Function）
函数是在数据库中完成特定任务的小型程序。它接受输入参数、执行计算并返回输出结果。

### 3.2.10 触发器（Trigger）
触发器是一个响应某些事件的数据库操作。它通常是一个存储过程，当满足某个条件时，自动执行该存储过程。

### 3.2.11 序列（Sequence）
序列是一个整数计数器，它用来生成唯一的数字。

## 3.3 Oracle的运行环境
### 3.3.1 物理结构
Oracle数据库具有三种物理结构：独立数据库实例、叶子和客户机/服务器数据库实例、统一资源数据库(URDB)。

#### （1）独立数据库实例
这是Oracle的最原始形式，只有一个实例，完全独立运行。它可以在单个服务器上安装，也可以跨越多个服务器分布部署。

#### （2）叶子和客户机/服务器数据库实例
这种结构，又称为分布式数据库实例，其由一个集中管理服务器（简称Master Server）、多个实例服务器（简称Instance Servers）、一个或多个备库（简称Standby Servers）组成。Master Server负责管理整个数据库系统，如启动、关闭、宕机恢复等，同时将请求路由到相应的Instance Server。Instance Servers负责实际的数据读写操作，可以部署在同一个服务器上，也可以部署在不同的服务器上。Standby Servers是一种冗余的服务器，可以减少数据库的单点故障，并提供数据热备份。

#### （3）统一资源数据库(URDB)
这是一种特殊的数据库，其结构由一个主实例服务器、一个或多个连接池服务器、一个共享的服务库组成。主实例服务器负责管理整个数据库系统，如启动、关闭、宕机恢复等，同时将请求路由到相应的连接池服务器。连接池服务器是一个特殊的服务器，它们负责接收客户端的连接请求，并建立到主实例服务器的连接。共享的服务库包括数据库引擎、存储过程、例程、包和网络资源等。URDB能够充分利用资源，并降低资源损失，是一种高效的数据库结构。

### 3.3.2 操作系统支持
Oracle数据库支持多种操作系统平台，包括Windows、Linux、Solaris、AIX、HP-UX、IRIX、Unix、OS/400和其他多种平台。

### 3.3.3 Java支持
为了与Java应用程序兼容，Oracle数据库提供了JDBC驱动程序，可以使用Java编写Oracle数据库应用程序。

### 3.3.4 可用性模型
Oracle数据库有三种可用的模型：数据库实时可用性（Real Time Availability）、主备可用性（Active-Standby Availability）和AlwaysOn可用性（Always On Availability）。

#### （1）数据库实时可用性
这是最简单的可用性模型，它只能提供最终一致性。

#### （2）主备可用性
这种结构，由一个主库和一个备库组成，其主库承担数据库读写操作，备库在主机出故障时，提供数据的热备份。主备可用性可以保证数据库的高可用性，但不能提供实时数据。

#### （3）AlwaysOn可用性

## 3.4 Oracle的数据库术语
### 3.4.1 物理文件
物理文件是数据库中用来存储数据的实际文件。它包括数据文件、日志文件、控制文件和字典文件等。

#### （1）数据文件
数据文件用来存储数据库中的表和索引数据。每张表对应于一个数据文件，表的每条记录占据一页的空间。

#### （2）日志文件
日志文件用来记录对数据库的操作。日志文件包括Redo（重做）日志、Undo（撤销）日志、归档日志等。

#### （3）控制文件
控制文件用来存储数据库的参数设置。

#### （4）字典文件
字典文件用来存储数据库中所有对象的定义。

### 3.4.2 数据字典
数据字典是由一系列的元数据组成，它反映了数据库中的对象和关系。数据字典包含所有数据库对象的定义，包括表、视图、字段、索引、存储过程、触发器、序列等。

### 3.4.3 目录结构
目录结构是指数据库物理文件在磁盘上存放的位置。它由数据库文件、控制文件的存放路径、用户及权限的存放路径等组成。

### 3.4.4 物理段
物理段（Physical Segment）是物理存储器中的连续区域。一个数据文件包含若干物理段。

### 3.4.5 数据块
数据块（Data Block）是用来存储数据的一个固定大小的内存页。它通常与物理段相对应。一个数据块通常与一个磁盘块对应。

### 3.4.6 数据页面
数据页面（Data Page）是用来存储数据的一个固定大小的磁盘页。它通常与数据块相对应。一个数据页面通常包含多个数据块。

### 3.4.7 索引组织结构
索引组织结构（Index Organized Table）是指在表里面创建一个逻辑的索引顺序，以便快速定位数据。在一个索引组织结构的表里，所有的记录都按照其索引列的顺序存放在磁盘上。

### 3.4.8 数据加密
数据加密（Encryption）是指对数据进行加密，以防止未经授权的访问。数据加密的方法有多种，包括用于数据库的加密密钥管理、表空间加密、块级数据加密等。

### 3.4.9 两阶段提交（Two Phase Commit）
两阶段提交（Two Phase Commit，2PC）是分布式数据库中使用的一种事务机制，它确保一个事务的原子性，并允许分布式事务中的参与者尽可能保持一致性。

## 4.3 MySQL的运行环境
### 4.3.1 物理结构
MySQL数据库是关系型数据库管理系统，它的物理结构一般包括三个层次：

- 服务层（Server Layer）：负责接收和处理客户端的请求；
- 存储引擎层（Storage Engine Layer）：负责数据的物理和逻辑存放；
- 物理层（Physical Layer）：负责数据的读写操作。

### 4.3.2 操作系统支持
MySQL数据库支持多种操作系统平台，包括Windows、Linux、Solaris、AIX、HP-UX、IRIX、Unix、OS/400和其他多种平台。

### 4.3.3 Java支持
为了与Java应用程序兼容，MySQL数据库提供了JDBC驱动程序，可以使用Java编写MySQL数据库应用程序。

## 4.4 MySQL的数据库术语
### 4.4.1 存储引擎
存储引擎（Storage Engine）是MySQL数据库的核心组件之一。它负责数据的物理和逻辑存放。MySQL共有几种存储引擎，如MyISAM、InnoDB、Memory等。

### 4.4.2 InnoDB
InnoDB是MySQL数据库的默认存储引擎，它提供了具有提交、回滚、崩溃恢复能力的事务安全机制，具有众多的性能优势，并且也支持行级锁定和外键。

### 4.4.3 MyISAM
MyISAM是MySQL数据库的第二代存储引擎。它没有全局事务的 ACID 属性，也没有外键。它适用于高效的SELECT、INSERT、UPDATE和DELETE操作，但不支持事务。

### 4.4.4 Memory
Memory是一种临时存储引擎，它所有的内容都存放在内存中。它的作用是作为缓存来加速数据库访问。

### 4.4.5 数据库模式
数据库模式（Schema）是用来定义数据库对象（如表、视图、触发器、存储过程等）的结构和约束的语法。

### 4.4.6 行锁与表锁
行锁与表锁是mysql数据库的两种最基本的锁，它用来帮助数据库事务正确处理并发问题。行锁是针对当前行的锁，对一行的数据进行访问加锁。表锁是对整个表的锁，对表中的所有数据进行访问加锁。

### 4.4.7 事务
事务（Transaction）是数据库的一个操作序列，它是一组数据库操作，要么全做，要么全不做。事务具有4个属性：原子性、一致性、隔离性、持久性。

### 4.4.8 锁
锁（Lock）是mysql数据库的重要机制，它用来确保事务之间的并发执行不会导致数据不一致。锁可以分为共享锁和排他锁，它可以阻止其他事务在相同资源上发生并发修改。

# 5.HBase数据库
## 5.1 HBase的介绍
Apache HBase (TM)是一个开源的分布式 NoSQL 数据库。它在 Hadoop 项目中作为 Hadoop Distributed File System (HDFS) 的补充来提供高容错性、高可靠性以及高性能的数据存储。

HBase 是 Apache Software Foundation 下的一个开源项目，它在 Google 的 GFS 上进行了改进，是一种分布式、可伸缩、能够在大数据量下提供高性能的数据存储的数据库。

HBase 主要提供以下几个方面的特性：

- 分布式数据存储：HBase 采用 Hadoop 文件系统接口（Hadoop FileSystem API）作为底层数据存储，结合 BigTable 论文提出的一些优化方法来实现分布式数据存储。
- MapReduce 友好型接口：HBase 提供 MapReduce 框架接口，使得 MapReduce 程序可以直接操纵 HBase 数据，大大方便了 Hadoop 生态圈的开发者使用。
- RESTful Web Service：HBase 通过 RESTful Web Services 来暴露自己的数据接口，方便其他系统进行接入。
- 高可用性和高扩展性：HBase 有主从架构，主节点负责处理客户端请求，而从节点可以提供快速的查询能力。HBase 可以横向扩展，在集群内动态添加节点，实现集群的无缝扩展。

## 5.2 HBase的基本概念
### 5.2.1 Namespace
命名空间（Namespace）是 HBase 中一个重要的抽象概念，它是 HBase 表的逻辑容器。命名空间在逻辑上类似于关系型数据库中的 Schema，用于对表进行分类管理。

### 5.2.2 Table
表（Table）是 HBase 的基本单元，它与关系型数据库中的表概念相似，是一个二维表结构。表的每一行都是一个 Cell，Cell 又可以看作是一个单元格。

### 5.2.3 RowKey
RowKey（行键）是 HBase 中一个重要的属性，它是每一行的唯一标识。

### 5.2.4 ColumnFamily
列族（ColumnFamily）是 HBase 中一个重要的抽象概念，它是用来对列进行分类管理的一种方式。列族可以理解成关系型数据库中的字段。

### 5.2.5 Qualifier
Qualifier（限定符）是 HBase 中一个重要的属性，它是列的第二级标识，与列族配合确定唯一的一个列。

### 5.2.6 Cell
单元格（Cell）是 HBase 中的最小存储单元，它是 HBase 表中的数据单元，由 RowKey、ColumnFamily、Qualifier、Timestamp 以及 Value 组成。

### 5.2.7 Timestamp
时间戳（Timestamp）是 HBase 中一个重要的属性，它可以记录数据写入的时间。

### 5.2.8 Value
值（Value）是 HBase 中一个重要的属性，它是 Cell 中的最主要属性，它存储的数据实际上是用户想要存储的数据。

## 5.3 HBase的特性
### 5.3.1 高可用性
HBase 在设计之初就考虑了高可用性。它采用 Master/Slave 架构，Master 节点主要处理客户端请求，而 Slave 节点提供快速的查询能力。HBase 会自动处理服务器宕机的情况，确保数据的可靠性。

### 5.3.2 列族数据存储
HBase 是按照列族（ColumnFamily）的方式来存储数据的，它支持多种类型的列族，可以根据需求对数据进行分组。

### 5.3.3 稀疏索引
HBase 提供了稀疏索引功能，可以减少索引的体积，提升查询效率。

### 5.3.4 Region 分片
HBase 是按照 Region（数据分片）的方式来存储数据的。Region 的大小可以通过配置文件进行调整，默认情况下，HBase 会把数据平均分成 128 个 Region。

### 5.3.5 线性查询
HBase 支持索引和全局扫描两种查询方式，线性查询可以快速定位数据。

### 5.3.6 数据一致性
HBase 是分布式数据库，它支持多个 Region 之间的数据同步，确保数据的一致性。

### 5.3.7 高性能
HBase 通过切分数据、使用批量写操作、利用内存映射和异步 IO 等手段，实现了高性能。

## 5.4 HBase的适用场景
### 5.4.1 实时数据存储
HBase 适用于实时数据存储，它提供高性能、高可靠性的数据存储功能。

### 5.4.2 小数据量存储
HBase 适用于小数据量的存储，它对数据进行分布式存储，可以支持 PB 级别的数据量。

### 5.4.3 大数据量分析
HBase 适用于大数据量的分析场景，它支持海量数据的并行查询和统计分析，在一定时间内可以完成复杂的查询操作。

# 6.MongoDB数据库
## 6.1 MongoDB的介绍
MongoDB 是一种基于分布式文件存储的 NoSQL 数据库。

它是一个开源的数据库软件，是一个快速、轻量级、无SCHEMA的文档型数据库。由于其分布式的存储特性，MongoDB 适合存储大量的复杂数据。

## 6.2 MongoDB的基本概念
### 6.2.1 文档（Document）
文档（Document）是 MongoDB 的基本数据结构。一个文档类似于 JSON 对象，由键-值对组成。

### 6.2.2 集合（Collection）
集合（Collection）是 MongoDB 中相对文档型数据库的另一个概念。集合是文档的容器，包含多个文档。

### 6.2.3 数据库（Database）
数据库（Database）是 MongoDB 中用于存储集合的容器。数据库相当于关系型数据库中的数据库。

### 6.2.4 连接（Connection）
连接（Connection）是指 MongoDB 数据库服务器端和客户端之间创建的连接。

### 6.2.5 查询指令（Query Instruction）
查询指令（Query Instruction）是 MongoDB 中用于查询数据库的命令。

## 6.3 MongoDB的适用场景
### 6.3.1 高性能实时数据存储
MongoDB 是一种高度优化的 NoSQL 数据库，它支持高性能、高可用性的数据存储。

### 6.3.2 移动应用程序数据存储
MongoDB 非常适合于移动应用程序的数据存储，因为它支持实时查询和实时更新。

### 6.3.3 社交媒体数据存储
MongoDB 非常适合于社交媒体数据存储，因为它支持数据查询和实时更新。

### 6.3.4 网站访问日志数据存储
MongoDB 可以用来存储网站访问日志数据，因为它支持高性能的数据存储和查询。

## 6.4 MongoDB的优缺点
### 6.4.1 优点
- 性能优异：基于分布式文件存储、支持高性能数据存储。
- 丰富的数据类型：支持丰富的数据类型，如字符串、数字、日期、数组、对象。
- 模型自由：集合中的文档不需要有相同的字段，即使不同文档具有相同的字段也是允许的。
- 索引功能：支持动态的索引功能，无需提前创建索引。
- 自动分片：数据自动分片，可以动态增加或减少机器。

### 6.4.2 缺点
- 复杂查询难度增大：由于有复杂的查询功能，使得复杂查询变得更加困难。
- 需要手动监控和管理：虽然 MongoDB 有自身的数据库监控工具，但还是需要人为的去管理。
